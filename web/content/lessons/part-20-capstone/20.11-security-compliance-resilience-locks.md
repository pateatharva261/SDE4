After this lesson you will be able to:

- Operationalize **compliance controls** (15.8): audit, encryption, secrets, PII handling, transaction monitoring.

## 2. Motivation

For a regulated financial platform, **security and compliance are existential**, and **resilience protects the money paths** from cascading failure. These are **cross-cutting concerns** — they apply across every context (20.1), not to one service. This lesson consolidates the protective machinery: how the platform stays **secure, compliant, resilient, and coordinated** under attack, load, and partial failure.

## 3. The design (framework — 1.3.1)

### 3.1 Security — defense in depth + zero-trust (Part 15)

`[BP]`
- **Zero-trust** (15.5): never trust the network; **authenticate + authorize every request by identity** (20.3), **mTLS** between all services (service mesh — 12.7), micro-segmentation, least privilege (15.1).
- **Encryption** (15.4): TLS everywhere (incl. internal); **at-rest** encryption for the ledger, documents, backups; **field-level/tokenization** for the most sensitive data (card data — PCI, PII).
- **Secrets** (15.4): a **secrets manager** (least-privilege, audited, rotated, short-lived dynamic secrets); keys in **KMS/HSM**; never commit secrets.
- **Threat modeling** (15.1 — STRIDE): applied per context; minimize attack surface; **assume breach**.
- **OWASP defenses** (15.6): parameterized queries (injection), output encoding, SSRF protection, and — the #1 risk — **broken access control** enforced server-side (20.3).
- `[BP]` **Security = zero-trust + defense-in-depth + encryption + secrets management + OWASP hygiene, applied across every context.**

### 3.2 Compliance controls (15.8)

`[BP]` Operationalizing the 20.1 compliance landscape:
- **Immutable audit trail** (20.7): every financial + sensitive action recorded, tamper-evident, redacting secrets — satisfies SOX/audit (15.8).
- **KYC/AML** (20.3): identity verification + **transaction monitoring** (stream/rules over transaction events — Part 9) flagging suspicious activity → compliance review.
- **PCI-DSS** (15.8/19.2.3): tokenize card data, minimize scope, isolate payment handling.
- **GDPR/privacy** (15.8): PII minimization, data-subject rights (erasure via **crypto-shredding** against the immutable ledger), residency (multi-region data placement — 20.10).
- **Data classification + governance:** know where PII/financial data lives; control access (20.3); retention policies.
- `[BP]` **Compliance = audit + KYC/AML monitoring + PCI + GDPR + governance — built into the architecture (20.1), not bolted on.**

### 3.3 Rate limiting (15.7 / 19.1.2)

`[BP]`
- **Dual purpose** (15.7): **reliability** (protect capacity — 11.4) + **security** (throttle abuse: brute-force, credential-stuffing, scraping, DoS).
- **Where:** at the **API gateway/edge** (20.9/12.6) — earliest, cheapest (load shedding — 11.4) — plus per-service defense in depth.
- **How** (19.1.2): **token bucket** + a **shared atomic counter (Redis — 6.6)** for global enforcement across instances; dimensions per-user/IP/API-key/endpoint (15.7); **429 + Retry-After** for backoff (9.9).
- **Financial specifics:** stricter limits on sensitive operations (login, money movement); combine with **MFA/lockout/anomaly detection** for auth abuse (15.7/20.3).
- `[BP]` Reuses **19.1.2** wholesale — the platform's abuse + capacity guardrail.

### 3.4 Resilience — circuit breakers, bulkheads, timeouts (11.3)

`[BP]`
- **Every remote call** (cross-service, external banking/market feeds) wrapped in **timeout + retry (backoff+jitter) + circuit breaker + bulkhead** (11.3) → prevents **cascading + metastable failures** (a slow dependency shouldn't topple the platform).
- **Bulkheads** (11.3): isolate resource pools so one failing dependency (e.g., a slow market-data vendor) can't exhaust threads platform-wide.
- **Graceful degradation** (11.4): if a non-critical dependency (recs, AI — 20.8) fails, **degrade** (hide the feature) rather than fail the whole request; **fail-closed** on money-critical checks (security > availability there — 15.1).
- `[BP]` **Resilience patterns (11.3/11.4) on every dependency** — especially around external rails + the money path.

### 3.5 Distributed locks (19.2.5) — only where needed

`[BP]`
- **Prefer lock-free** (19.2.5): idempotency (11.5), partitioning (single-writer-per-key — 7.3), and CAS handle most coordination **without** distributed locks.
- **When needed** (leader election, exclusive resource access, dedup of a singleton job): use a **correct** lock — **lease (TTL) + fencing token + consensus store (etcd/ZooKeeper — CP)** (19.2.5/8.3.6) — with the **resource enforcing the fencing token** (safety). Distinguish **efficiency vs correctness locks** (19.2.5).
- **Example:** ensuring a scheduled reconciliation/settlement job runs on exactly one node → a fenced lease-based lock (or a partitioned assignment).
- `[BP]` **Locks are a last resort; when used, do them correctly (lease + fencing + consensus — 19.2.5).**

### 3.6 Deep dives + bottlenecks

`[BP]`
- **Cross-cutting enforcement:** security/compliance/resilience apply **per context** — the service mesh (12.7) + gateway (20.9) + shared libraries standardize them.
- **Fail-open vs fail-closed** (15.1/11.4): rate limiter usually fail-open (don't break the API); **money/security checks fail-closed** (deny on doubt) — a deliberate per-control choice.
- **Compliance as ongoing:** monitoring + audit are continuous, not one-time.
- **Bottleneck risks:** the shared rate-limit counter (HA/shard — 19.1.2), the lock/consensus service (HA — 19.2.5), the secrets manager (cache carefully) — all critical dependencies to make HA.
- `[BP]` **The lesson:** cross-cutting protection = **zero-trust security + defense-in-depth + encryption/secrets (Part 15) + operational compliance (audit/KYC-AML/PCI/GDPR — 15.8) + dual-purpose rate limiting (15.7/19.1.2) + resilience patterns on every dependency (11.3/11.4) + correct distributed locks only where needed (19.2.5)** — applied across every context, with money/security controls **failing closed** and reliability controls **failing open**.

## 5. Real-World Analogy

Think of a **bank's guards, compliance office, crowd control, safety valves, and the single key to the vault schedule**.

- **Security (zero-trust) = ID checks at every door, not just the entrance:** guards verify identity **everywhere inside**, cameras and sealed rooms limit damage if someone gets in (assume breach), and the safe combinations are held by a **tightly-controlled, rotating key service**.
- **Compliance = the compliance office watching transactions + keeping unerasable records:** it verifies customers (KYC), watches for suspicious money flows (AML), and keeps an **immutable logbook** auditors can inspect.
- **Rate limiting = crowd control at the doors:** it stops a mob (DoS) or someone trying thousands of PINs (brute-force) from overwhelming the branch — and tells turned-away people **when to come back** (Retry-After).
- **Circuit breakers/bulkheads = safety valves + compartments:** if the plumbing to one partner bank backs up, a **valve trips** so the pressure doesn't burst every pipe, and **watertight compartments** stop one flooded room from sinking the whole ship.
- **Distributed lock = the single vault-schedule key:** only used for the rare "exactly one person may do this at a time" job — and when used, it's a **checked-out badge with a serial number** the vault verifies (fencing), so a forgotten stale badge can't reopen the vault.
- **Fail-closed vs fail-open:** if the guard's scanner breaks, the **vault stays shut** (money: deny on doubt), but the **information desk stays open** (reads: don't punish everyone for a broken scanner).

## 7. Implementation Details

- **Security:** zero-trust + mTLS (12.7), encryption in transit/at-rest + tokenization (15.4), secrets manager/KMS (15.4), OWASP defenses + server-side authz (15.6/20.3) (§3.1).
- **Compliance:** immutable audit (20.7), KYC/AML monitoring (Part 9), PCI tokenization, GDPR crypto-shredding/residency, governance (15.8) (§3.2).
- **Rate limiting:** token bucket + shared atomic Redis at gateway + per-service (15.7/19.1.2); stricter on sensitive ops; 429+Retry-After (§3.3).
- **Resilience:** timeout/retry+jitter/circuit-breaker/bulkhead on every remote call (11.3); graceful degradation (non-critical) + fail-closed (money) (11.4) (§3.4).
- **Locks:** prefer lock-free (idempotency/partitioning/CAS); when needed, lease+fencing+consensus (19.2.5) (§3.5).

## 8–14. (Condensed)

**Advantages:** layered protection (defense in depth); existential security/compliance covered; cascade prevention (resilience); abuse + capacity guardrails; correct coordination where needed.
**Disadvantages/cautions:** cross-cutting machinery adds latency/complexity + critical dependencies (rate-limit counter, lock service, secrets manager — make HA); fail-open/closed must be chosen deliberately; compliance is ongoing cost.
**When NOT to:** don't fail-open on money/security checks; don't use distributed locks where idempotency/partitioning suffices; don't treat compliance as one-time.
**Common mistakes:** trusting the internal network (no mTLS); client-side authz (broken access control); missing circuit breakers (cascades); per-IP-only rate limiting (NAT/botnets); incorrect locks (no fencing — 19.2.5); logging secrets/PII.
**Interview Qs:** 🟢 What does zero-trust mean here? 🟡 Rate limiting — reliability or security? Where + how? 🔴 How do you prevent cascading failures (breakers/bulkheads)? When do you use a distributed lock, and how correctly? ⚫ Full cross-cutting design: security, compliance, rate limiting, resilience, locks — and where each fails open vs closed.
**Production pitfalls:** lateral movement (no zero-trust); cascade from a slow dependency (no breaker); rate-limit counter outage (fail-open); stale-lock split-brain (no fencing); compliance gaps; secret leakage.
**Optimizations:** service mesh for uniform mTLS/policy (12.7); shared resilience libraries; HA sharded rate-limit counter (19.1.2); prefer lock-free (11.5/7.3); cache authz/secrets carefully; automated compliance monitoring.

## 15. Summary

For a regulated financial platform, **security and compliance are existential** and **resilience protects the money paths** — all **cross-cutting concerns** applied across every context (20.1), consolidated here. **Security** is **zero-trust + defense-in-depth** (15.5): never trust the network, **authenticate + authorize every request by identity** (20.3), **mTLS** between all services (mesh — 12.7), micro-segmentation, least privilege; **encryption everywhere** (TLS incl. internal, at-rest for ledger/documents/backups, **field-level/tokenization** for card data/PII — 15.4); **secrets** in a rotated, audited, least-privilege manager with keys in **KMS/HSM** (15.4); **threat modeling (STRIDE)** per context assuming breach; and **OWASP defenses** (parameterized queries, output encoding, SSRF protection, and server-side enforcement of the #1 risk — **broken access control** — 15.6/20.3). **Compliance** is **operationalized** (15.8): an **immutable audit trail** (20.7 — SOX), **KYC/AML** identity verification + **transaction monitoring** (stream/rules over events — Part 9), **PCI-DSS** tokenization + scope minimization (19.2.3), **GDPR** PII minimization + erasure via **crypto-shredding** + residency (multi-region placement — 20.10), and data classification/governance — **built into the architecture, not bolted on**. **Rate limiting** (15.7, reusing **19.1.2**) serves **dual purposes** — reliability (protect capacity — 11.4) + security (throttle brute-force/stuffing/scraping/DoS) — enforced at the **gateway/edge** (earliest/cheapest — 20.9) plus per-service, via **token bucket + a shared atomic Redis counter** for global enforcement, with stricter limits on sensitive operations (login/money) combined with MFA/lockout/anomaly detection, and **429 + Retry-After** for backoff. **Resilience** wraps **every remote call** (cross-service + external banking/market rails) in **timeout + retry(backoff+jitter) + circuit breaker + bulkhead** (11.3) to prevent **cascading/metastable failures**, with **graceful degradation** for non-critical dependencies (recs/AI — 20.8 hidden on failure) but **fail-closed** on money-critical checks (security > availability there). **Distributed locks** are a **last resort** — prefer **lock-free** designs (idempotency — 11.5, single-writer partitioning — 7.3, CAS) — but when genuinely needed (leader election, singleton reconciliation/settlement jobs) use a **correct** lock: **lease (TTL) + fencing token + consensus store (etcd/ZooKeeper — CP)** with the **resource enforcing the fencing token** (19.2.5/8.3.6), distinguishing efficiency vs correctness locks. The **key operational judgment** is **fail-open vs fail-closed** (15.1/11.4): the **rate limiter usually fails open** (a broken limiter shouldn't break the API) while **money/security checks fail closed** (deny on doubt) — a deliberate per-control choice — and the critical shared dependencies (rate-limit counter, lock/consensus service, secrets manager) must all be **HA**. In one line: **zero-trust security + defense-in-depth + encryption/secrets + operational compliance (audit/KYC-AML/PCI/GDPR) + dual-purpose rate limiting + resilience patterns on every dependency + correct locks only where needed — money/security fail-closed, reliability fail-open**.
