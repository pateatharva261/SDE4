## 1. Learning Objectives

After this lesson you will be able to:

- Design the **edge + delivery + caching + object-storage** layers of the platform (the read-performance + static-asset infrastructure).
- Place **caching** (Part 6) correctly: cache **derived read models** (portfolio views, market prices), **never** authorize money off a cache (20.4).
- Use **object storage** (4.3.2) for documents/statements/reports and a **CDN** (18.4) for static + cacheable content.
- Design the **API gateway + BFF** (12.6) as the single edge: routing, authN, rate limiting, aggregation.
- Handle deep dives: **cache invalidation** for financial data, **what's cacheable vs not**, and **secure document delivery**.

---

## 2. Motivation

The platform is **read-heavy** (portfolio dashboards — 20.2), serves **large static/document content** (statements, tax docs, reports), and needs a **clean, secure edge** for many client types (web/mobile/advisor/API). This lesson assembles the **caching (Part 6), object storage (4.3.2), CDN (18.4), and API gateway (12.6)** layers — the infrastructure that makes reads fast and delivery efficient **without compromising the correctness of money** (the golden rule: cache derived views, never the authoritative ledger for authorization).

## 3. The design (framework — 1.3.1)

### 3.1 Caching (Part 6) — what and where

`[CS]` Cache **derived, read-mostly** data; be careful with money `[BP]`:
- **Cache these:** portfolio **read-model** views (positions/balances/valuations — 20.4/20.7), **latest market prices** (20.5), search results (20.8), reference data (instrument metadata), session/authz decisions (carefully — 20.3).
- **Topologies (6.2):** CDN/edge (static) → API/gateway cache → **distributed cache (Redis — 6.6)** for shared read models → local caches. Cache-aside (6.3) for most read models.
- **The golden rule** (20.4): **never authorize a money operation off a cached balance** — reads for **display** can be cached (eventually consistent — 10.5), but **authorization/execution checks the authoritative ledger**. Confusing display-cache with truth is a correctness bug.
- **Invalidation** (6.5 — one of the hard things): financial read models update from **events** (CDC — 9.8) → **event-driven invalidation/refresh** of caches (not just TTL) so a trade reflects quickly; use short TTLs + versioned keys for prices. Balance freshness (users expect near-real-time portfolios) against load.
- `[BP]` **Cache derived views aggressively (Part 6), invalidate via events (9.8), never cache the authorization truth.**

### 3.2 Object storage (4.3.2)

`[BP]`
- **Documents/statements/tax reports/uploads/KYC docs** → **object/blob storage** (4.3.2 — S3-style): cheap, durable, scalable, lifecycle-managed (tiering hot→cold→archive — 4.3.2).
- **Immutable + versioned** where needed (statements are records — audit); **encrypted at rest** (15.4); **access-controlled** (a client's docs are private — 20.3).
- **Secure delivery:** serve via **signed, expiring URLs** (not public buckets) so only authorized users fetch a document — a common, important pattern.
- `[BP]` **Object storage for documents/reports** — durable, tiered, encrypted, access-controlled, delivered via signed URLs.

### 3.3 CDN & edge (18.4)

`[BP]`
- **CDN** (18.4/3.3.3) for **static assets** (app bundles, images) and **cacheable content** (public reference data, and cached documents via signed URLs at the edge).
- **Edge benefits:** low global latency, origin offload, **DDoS absorption + WAF** (18.4/15.5) — the security first tier (20.11).
- **Careful:** **personalized/sensitive financial data is NOT edge-cached** publicly — only static/public or per-user-authorized (signed) content. Never cache a portfolio at a shared edge.
- `[BP]` **CDN for static + public + authorized-document delivery + DDoS/WAF edge** — not for private financial views.

### 3.4 API gateway + BFF (12.6)

`[BP]`
- **API gateway** (12.6): the **single edge** for all client traffic — **routing** to services, **authN/token validation** (20.3), **rate limiting** (15.7/20.11), request/response shaping, and observability (20.12). North-south entry.
- **BFF (Backend-for-Frontend)** (12.6): per-client-type gateways (web/mobile/advisor/public-API) that **aggregate** downstream calls into client-shaped responses (e.g., a dashboard = portfolio + prices + recs in one call), reducing round-trips (17.4) + tailoring payloads.
- **Load balancing** (3.3.1) in front; TLS termination at the edge (15.4).
- `[BP]` **Gateway = single secure edge (routing/authN/rate-limiting); BFF = client-tailored aggregation.**

### 3.5 Deep dives + bottlenecks

`[BP]`
- **Cache correctness** (§3.1): display-cache vs authorization-truth — the golden rule (20.4); event-driven invalidation (9.8) for freshness.
- **Read scaling** (7.5): cache + read replicas dissolve the read-heavy load (20.2); the gateway/BFF must scale statelessly (7.2).
- **Secure delivery:** signed expiring URLs for documents; no public buckets; no edge-caching private data.
- **Gateway as a critical path:** it's on every request → must be **HA + fast + not a bottleneck** (scale horizontally, avoid heavy logic); rate limiting protects backends (20.11).
- **Bottleneck:** without caching, the ledger/DB would be hammered by reads → **cache-first + replicas** (Part 6/7.5) is the dominant relief (20.2); object storage + CDN offload large content.
- `[BP]` **The lesson:** the read/delivery infrastructure = **cache derived views (Part 6, event-invalidated — never the auth truth) + object storage for documents (4.3.2, encrypted, signed-URL delivery) + CDN for static/public/DDoS-WAF edge (18.4) + API gateway/BFF as the single secure, aggregating edge (12.6)** — making reads fast and delivery efficient while keeping money correct.

## 5. Real-World Analogy

Think of a **bank's branch front, mailroom, and information boards**.

- **Caching = the info boards showing recent balances and prices:** they refresh **moments after** something changes, so customers get fast answers without a clerk digging in the vault each time. But — the golden rule — **the teller never pays out based on the board**; to actually move money, they **check the official ledger**. The board is for looking; the vault is for doing.
- **Object storage = the document archive/mailroom:** statements, tax forms, and uploaded IDs are filed in a **big, cheap, durable archive**, with old ones moved to deep storage. You don't get them by walking into the archive — you're handed a **single-use claim ticket** (a signed, expiring link) that only works for **your** documents.
- **CDN = neighborhood pickup lockers:** the bank's **public brochures and app** are stocked in lockers all over the city for instant pickup, and those lockers **absorb the crowd** (DDoS) at the edge. But your **private statement** is never left in a shared public locker — only static or personally-authorized items go to the edge.
- **API gateway + BFF = the front desk that routes you and bundles your errands:** one **secure entrance** checks your ID and directs you, and a helpful concierge **bundles several requests** ("here's your dashboard: balances, prices, and suggestions, all at once") so you don't queue at five counters.

## 7. Implementation Details

- **Caching:** distributed cache (Redis — 6.6) for derived read models + latest prices; cache-aside (6.3); **event-driven invalidation** (9.8); **never cache the authorization truth** (20.4) (§3.1).
- **Object storage** (4.3.2): documents/statements — encrypted (15.4), lifecycle-tiered, access-controlled (20.3), **signed expiring URLs** (§3.2).
- **CDN** (18.4): static + public + authorized-document edge delivery + WAF/DDoS (20.11); no private-data edge caching (§3.3).
- **API gateway + BFF** (12.6): routing, authN (20.3), rate limiting (20.11), aggregation; LB (3.3.1) + TLS termination; stateless scale (7.2) (§3.4).

## 8–14. (Condensed)

**Advantages:** fast read-heavy dashboards (cache + replicas); cheap durable document storage; low-latency static delivery + edge security; clean single secure edge + reduced round-trips (BFF).
**Disadvantages/cautions:** cache invalidation is hard (event-driven + TTL); risk of caching stale/authoritative data (golden rule); gateway is a critical path (HA); signed-URL + access-control complexity for docs.
**When NOT to:** don't cache/authorize money off a cache; don't edge-cache private financial data; don't put heavy logic in the gateway; don't serve documents from public buckets.
**Common mistakes:** authorizing trades off cached balances (correctness bug); public buckets leaking documents; edge-caching personalized data; no event-driven invalidation (stale portfolios); gateway as an overloaded monolith.
**Interview Qs:** 🟢 What do you cache and what do you never cache? 🟡 How do you invalidate financial read-model caches (events vs TTL)? 🔴 How do you deliver private documents securely (signed URLs, no public/edge)? Gateway vs BFF? ⚫ Design the full read/delivery infra: caching (with the golden rule), object storage, CDN, gateway/BFF — and defend the correctness boundaries.
**Production pitfalls:** stale portfolio caches; authorization off stale cache; document leakage (misconfigured buckets/URLs); gateway saturation; edge caching sensitive data.
**Optimizations:** event-driven cache invalidation (9.8); short-TTL + versioned keys for prices; read replicas (7.5); tiered object storage (4.3.2); BFF aggregation (17.4); edge WAF/DDoS (18.4).

## 15. Summary

The platform is **read-heavy** (portfolio dashboards — 20.2), serves **large document content** (statements, tax docs, KYC uploads), and needs a **clean secure edge** for many client types — so this lesson assembles the **caching (Part 6), object storage (4.3.2), CDN (18.4), and API gateway/BFF (12.6)** layers that make reads fast and delivery efficient **without compromising money correctness**. **Caching** targets **derived, read-mostly** data — portfolio **read-model** views (20.4/20.7), **latest market prices** (20.5), search results (20.8), reference data — across the standard topologies (CDN → gateway cache → distributed Redis — 6.6 → local), typically cache-aside (6.3); the **golden rule** (20.4) is **never authorize a money operation off a cached balance** — display reads can be cached (eventually consistent — 10.5) but **authorization/execution must check the authoritative ledger** (confusing display-cache with truth is a correctness bug) — and because financial read models update from **events**, use **event-driven invalidation/refresh** (CDC — 9.8), not just TTL, so trades reflect quickly. **Object storage** (4.3.2, S3-style) holds **documents/statements/tax reports/uploads** — cheap, durable, **lifecycle-tiered** (hot→cold→archive), **immutable/versioned** where records demand, **encrypted at rest** (15.4), **access-controlled** (20.3), and delivered via **signed, expiring URLs** (never public buckets). The **CDN** (18.4) serves **static assets + public/cacheable content + authorized documents (signed URLs)** at low global latency with **origin offload + DDoS absorption + WAF** (the security first tier — 20.11) — but **personalized/sensitive financial data is never edge-cached** publicly. The **API gateway** (12.6) is the **single secure edge** (routing, authN/token validation — 20.3, **rate limiting** — 15.7/20.11, observability — 20.12), fronted by a **load balancer + TLS termination** (3.3.1/15.4), and **BFFs** (per client type) **aggregate** downstream calls into client-shaped responses (a dashboard = portfolio + prices + recs in one call — reducing round-trips — 17.4). **Deep dives:** cache correctness (display vs auth-truth — the golden rule), read scaling (cache + replicas dissolve the read-heavy load — 7.5; gateway/BFF scale statelessly — 7.2), secure document delivery (signed URLs, no public/edge for private data), and the gateway as an **HA critical path**. The **bottleneck** — reads hammering the ledger/DB — is relieved by **cache-first + replicas** (the dominant lever — 20.2), with object storage + CDN offloading large content. In one line: **cache derived views (event-invalidated, never the auth truth) + object storage for documents (encrypted, signed-URL delivery) + CDN for static/public/DDoS-WAF edge + API gateway/BFF as the single secure aggregating edge**.

## 16. Revision Notes (flashcard-ready)

- **Q:** What do you cache vs never cache? **A:** Cache derived read views (portfolio, prices, search); never authorize money off a cache — check the authoritative ledger (20.4).
- **Q:** How invalidate financial caches? **A:** Event-driven (CDC — 9.8) refresh + short TTL/versioned keys — not TTL alone — so trades reflect quickly.
- **Q:** Where do documents live + how delivered? **A:** Object storage (4.3.2) — encrypted, tiered, access-controlled; delivered via signed, expiring URLs (no public buckets).
- **Q:** What goes on the CDN? **A:** Static assets + public/cacheable content + authorized docs (signed URLs) + WAF/DDoS; never private financial data at a shared edge.
- **Q:** API gateway role? **A:** Single secure edge — routing, authN, rate limiting, observability; HA critical path, keep it thin.
- **Q:** BFF? **A:** Per-client-type gateway that aggregates downstream calls into client-shaped responses (fewer round-trips — 17.4).
- **Q:** Dominant read-scaling lever? **A:** Cache-first + read replicas (Part 6/7.5) — dissolves the read-heavy portfolio load.
- **Q:** Golden rule? **A:** Cache for display, never for authorization; the ledger is the truth (20.4).
