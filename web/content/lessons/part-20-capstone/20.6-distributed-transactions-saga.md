## 1. Learning Objectives

After this lesson you will be able to:

- Design **cross-context business transactions** (a trade, a funded purchase, a transfer) that span **multiple services** in the platform (20.1).
- Explain why **2PC is avoided** (11.6 — blocking, fragile, CP) and **sagas** (11.7) are preferred for cross-service consistency.
- Design a concrete **trade/purchase saga** with **compensating transactions** and choose **orchestration vs choreography** (11.7).
- Solve the **dual-write problem** with the **transactional outbox** (12.5) and ensure steps are **idempotent** (11.5).
- Handle the lack of isolation (**semantic locks / reservations**) and reason about **eventual consistency** across contexts.

---

## 2. Motivation

Because the platform is decomposed into **bounded contexts with database-per-service** (20.1/12.4), a single business operation — e.g., **"buy $10,000 of a fund"** — spans **Payments, Portfolio, Ledger, and Compliance** contexts. There's **no cross-service ACID transaction** (12.4). Yet the operation must be **atomic in effect** (all-or-nothing) and **correct** (money — 20.4). This is the **saga** problem (11.7), applied to the platform's most important flows.

---

## 3. The design (framework — 1.3.1)

### 3.1 Why not 2PC (11.6)

`[BP]`
- **Two-phase commit (2PC)** gives cross-service atomicity but is **blocking** (a coordinator or participant crash can leave others locked — the blocking problem — 11.6), **fragile/CP**, and **couples availability** of all participants. It also doesn't work with **external systems** (banking rails don't join your 2PC).
- `[BP]` **Avoid 2PC** for cross-service business transactions (11.6). Use **sagas** — local transactions + compensations — which trade **atomicity + isolation** for **availability + autonomy** (11.7).

### 3.2 The saga (11.7)

`[CS]` A **saga** = a sequence of **local transactions**, each in one service/context, where a failure triggers **compensating transactions** that semantically undo prior steps `[BP]`:
- Each step commits **locally** (its own ACID transaction — 12.4) and emits an event / calls the next step.
- On failure, run **compensations in reverse** (e.g., refund a charge, release a reservation, reverse a ledger entry via a new compensating entry — 20.4).
- **No isolation** (11.7): intermediate states are visible → use **semantic locks / reservations** (mark funds "reserved," an order "pending") to prevent conflicting concurrent operations, and design compensations to be **safe + idempotent** (11.5).
- `[BP]` **Saga = local txns + compensations, eventual consistency, no isolation → semantic locks + idempotent compensations.**

### 3.3 Example: the "funded purchase" saga

`[BP]` Buying $10,000 of a fund:
1. **Compliance check** (suitability/AML — 20.3/20.11) — validate the order is allowed.
2. **Reserve funds** (Payments/Portfolio) — a **reservation** (semantic lock), compensation = release.
3. **Execute the order** (route to broker/matching — possibly external) — compensation = cancel/reverse if not filled.
4. **Post to the ledger** (double-entry, immutable — 20.4) — compensation = compensating entry.
5. **Update the portfolio read model** + **notify** (eventual — 20.4/19.1.4).
- If any step fails, **compensate** the completed prior steps in reverse. **Irreversible steps last** (11.7) — do reversible/internal steps before the hard-to-undo external execution where possible.

### 3.4 Orchestration vs choreography (11.7)

`[BP]`
- **Orchestration:** a central **saga orchestrator** drives the steps + compensations. **Pros:** explicit, auditable, easy to reason about + monitor (good for **complex financial flows** — you want a clear, traceable state machine). **Cons:** the orchestrator is a component to build/own.
- **Choreography:** services react to each other's events, no central coordinator. **Pros:** decoupled. **Cons:** the flow is **emergent** (hard to follow/debug), risky for complex money flows.
- `[BP]` **Recommendation:** **orchestration for the money-critical sagas** (trades, purchases, transfers) — the auditability + explicit state machine matter for finance + compliance (20.7 audit). Choreography is fine for simpler, looser event reactions (e.g., notifications).

### 3.5 Dual writes + idempotency (12.5 / 11.5)

`[BP]`
- **Dual-write problem:** a step must **update its DB and emit an event/trigger the next step atomically** — but a DB commit + a separate message publish isn't atomic (crash between them → lost/duplicate). Solve with the **transactional outbox** (12.5): write the state change + an outbox row in **one local transaction**; a relay publishes the event (or CDC tails it — 9.8). **No dual writes.**
- **Idempotency everywhere** (11.5): steps + compensations may be retried (at-least-once — 9.4) → dedup by keys so retries are safe → **exactly-once effects**. Essential for saga correctness.
- `[BP]` **Outbox (no dual writes) + idempotent steps/compensations** make the saga reliable + recoverable.

### 3.6 Deep dives + bottlenecks

`[BP]`
- **No isolation** (§3.2): semantic locks/reservations prevent anomalies (e.g., double-spending reserved funds); design for visible intermediate states.
- **Compensation correctness:** some steps are **hard/impossible to undo** (an executed market trade) → **irreversible-last** + reserve-then-commit patterns; where truly irreversible, compensate by a **corrective business action** (sell back, adjust).
- **Consistency:** the saga is **eventually consistent** across contexts (10.5) — acceptable for the cross-context flow, while each **local** step (esp. the ledger post) is **strongly consistent** (20.4). Invariant-bound data stays within one context/transaction (12.4).
- **Observability** (20.12): sagas need **tracing + state visibility** (which step, stuck sagas, compensation failures) — orchestration helps.
- **Bottleneck / risk:** long-running sagas + compensation edge cases; mitigate with timeouts, monitoring, and idempotent retries.
- `[BP]` **The lesson:** cross-context business transactions = **sagas (not 2PC) — local ACID steps + compensations, orchestrated for money-critical flows, with semantic locks (no isolation), the transactional outbox (no dual writes), and idempotent steps/compensations (exactly-once effects)** — eventually consistent across contexts, strongly consistent within the ledger step.

## 5. Real-World Analogy

Think of **booking a complex trip through a travel agent** — flight, hotel, car — where there's **no single "book everything at once" button**.

- **No 2PC = you can't freeze all three vendors while you decide:** the airline, hotel, and rental company are **independent businesses**; you can't hold them all in a locked limbo until every piece confirms. Trying to would leave everyone waiting and blocked.
- **Saga = book one at a time, with cancellation policies:** the agent books the flight, then the hotel, then the car — **each a committed step** — and if the car falls through, they **cancel the hotel and flight** (compensating actions) to unwind the trip.
- **Semantic locks = reservations:** while arranging, the agent **holds** a seat and a room (a reservation others can't grab) rather than leaving them open to double-booking.
- **Orchestration = the agent runs the checklist:** one coordinator drives the sequence and the cancellations, so there's a **clear, traceable record** of where the booking stands — much better for something important than each vendor just reacting to the others (choreography), where no one can say what state you're in.
- **Idempotency = a booking reference so retries don't double-book:** if the agent resends a request, the vendor recognizes the reference and doesn't book twice.
- **Irreversible-last = pay the non-refundable deposit only after everything else is locked in.**

## 7. Implementation Details

- **Avoid 2PC** cross-service (11.6); use **sagas** (11.7) (§3.1).
- **Orchestrated saga** for money-critical flows (trades/purchases/transfers): local ACID steps + compensations, irreversible-last (§3.3/3.4).
- **Semantic locks/reservations** for the no-isolation problem (§3.2); **transactional outbox** for no dual writes (12.5/9.8) (§3.5).
- **Idempotent** steps + compensations (11.5) → exactly-once effects (9.4) (§3.5).
- **Eventually consistent** across contexts (10.5); ledger step strongly consistent (20.4); trace + monitor sagas (20.12) (§3.6).

## 8–14. (Condensed)

**Advantages:** cross-service atomic-in-effect operations without blocking 2PC; autonomy + availability; auditable orchestrated flow; recoverable (idempotent + outbox); works with external systems.
**Disadvantages/cautions:** no isolation (semantic locks needed); eventual consistency across contexts; compensation logic is complex + must be idempotent; some steps hard to undo (irreversible-last); orchestrator to own.
**When NOT to:** don't use 2PC for cross-service business transactions; don't use choreography for complex money flows (unauditable); don't skip semantic locks or idempotency.
**Common mistakes:** attempting distributed ACID across services; dual writes (use outbox); non-idempotent compensations (double-undo); no reservations (concurrent conflicts); irreversible step first; unmonitored stuck sagas.
**Interview Qs:** 🟢 Why not 2PC across services? 🟡 What is a saga + compensating transaction? 🔴 Orchestration vs choreography for a trade — which and why? How do you handle no-isolation + dual writes? ⚫ Design the full funded-purchase saga with compensations, semantic locks, outbox, idempotency, and consistency reasoning.
**Production pitfalls:** compensation failures (retry/idempotency); stuck/long-running sagas (timeouts + monitoring); visible intermediate states causing anomalies; lost/duplicate events (outbox); irreversible-step ordering mistakes.
**Optimizations:** orchestrator with a durable state machine; idempotency keys per step; outbox + CDC (9.8); reservations with TTL; observability/tracing for saga state (20.12); irreversible-last ordering.

## 15. Summary

Because the platform is decomposed into **bounded contexts with database-per-service** (20.1/12.4), a single business operation — e.g., **"buy $10,000 of a fund"** — spans **Compliance, Payments, Portfolio, and Ledger** contexts with **no cross-service ACID transaction**, yet must be **atomic in effect and correct** (money — 20.4). **2PC is avoided** (11.6): it's **blocking** (coordinator/participant crashes leave others locked — the blocking problem), **fragile/CP**, couples all participants' availability, and doesn't work with **external systems** (banking rails don't join your 2PC). Instead, use a **saga** (11.7): a sequence of **local ACID transactions**, each in one context, where a failure triggers **compensating transactions** that semantically undo prior steps — trading **atomicity + isolation** for **availability + autonomy**. The concrete **funded-purchase saga**: compliance check → **reserve funds** (a semantic lock; compensation = release) → **execute the order** (external; compensation = cancel/reverse) → **post to the immutable ledger** (compensation = a compensating entry — 20.4) → update the portfolio read model + notify — with **irreversible steps last** (11.7) and **compensations in reverse** on failure. Because sagas have **no isolation** (intermediate states are visible), use **semantic locks / reservations** (reserved funds, pending orders) to prevent conflicting concurrent operations, and design compensations to be **safe + idempotent** (11.5). For **money-critical flows**, prefer **orchestration** (a central orchestrator drives steps + compensations — explicit, auditable, monitorable — essential for finance/compliance/audit — 20.7) over **choreography** (decoupled but emergent/hard-to-debug — fine only for simpler event reactions like notifications). Solve the **dual-write problem** with the **transactional outbox** (12.5): write the state change + an outbox row in one local transaction and let a relay/CDC publish the event (9.8) — **no dual writes** — and make every step + compensation **idempotent** (retries are safe → **exactly-once effects** — 9.4). The saga is **eventually consistent across contexts** (acceptable — 10.5) while each **local step (especially the ledger post) is strongly consistent** (20.4), keeping invariant-bound data within one context (12.4); sagas need **tracing + state observability** (20.12) to catch stuck sagas + compensation failures. In one line: **sagas not 2PC — local ACID steps + idempotent compensations, orchestrated for money flows, with semantic locks (no isolation) and the transactional outbox (no dual writes) — eventually consistent across contexts, strongly consistent in the ledger step**.

## 16. Revision Notes (flashcard-ready)

- **Q:** Why not 2PC across services? **A:** Blocking (crash locks participants), fragile/CP, couples availability, doesn't span external systems (11.6).
- **Q:** What is a saga? **A:** A sequence of local ACID transactions with compensating transactions on failure; eventual consistency, no isolation (11.7).
- **Q:** Handle the no-isolation problem? **A:** Semantic locks / reservations (reserved funds, pending orders) + idempotent compensations.
- **Q:** Orchestration vs choreography? **A:** Orchestration (central, explicit, auditable) for money flows; choreography (emergent) only for simple event reactions.
- **Q:** Dual-write problem + fix? **A:** DB commit + separate publish isn't atomic → transactional outbox (one local txn + relay/CDC) — no dual writes (12.5).
- **Q:** Why idempotency in sagas? **A:** Steps/compensations get retried (at-least-once) → dedup for exactly-once effects (11.5/9.4).
- **Q:** Irreversible-last? **A:** Do reversible/internal steps before hard-to-undo external ones (11.7).
- **Q:** Consistency of a saga? **A:** Eventually consistent across contexts; each local step (ledger) strongly consistent (20.4).
- **Q:** Compensating an executed trade? **A:** Can't literally undo → corrective business action (sell back/adjust) via a compensating ledger entry.
