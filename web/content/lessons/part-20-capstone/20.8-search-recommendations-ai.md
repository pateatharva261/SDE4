## 2. Motivation

Beyond core money movement, the platform must help users **find** things (search instruments, transactions, documents) and **decide** things (recommended allocations, insights). These are **derived, read-side, eventually-consistent** capabilities (20.7) — they consume the event stream + market data, never sit on the CP ledger path. They reuse **18.7** (search) and **18.5/19.2.7** (recs), but with a **financial twist**: advice is **regulated** (suitability/fiduciary — 15.8), so recommendations aren't just "engagement" — they must be **appropriate + explainable + compliant**.

## 3. The design (framework — 1.3.1)

### 3.1 Search (18.7)

`[BP]`
- **What's searched:** instruments (name/ticker/sector), transactions, holdings, documents/statements, help content.
- **Design:** an **inverted index** (18.7 — Elasticsearch/Lucene-style) with text analysis + relevance ranking (BM25) + filters (sector, date, account) + **geo/faceted** search as needed. Sharded + scatter-gather for scale (18.7).
- **As a derived read model (CQRS — 20.7):** the index is **built from events** (ledger events, instrument updates, document uploads) via **CDC/projections** (9.8) → **eventually consistent** (18.7's "derived read model" lesson). Losing/rebuilding the index = replay (20.7).
- **Access control in search:** results must respect **authorization** (a user searches only **their** transactions; an advisor only **their clients'** — ReBAC — 20.3) → filter by permission at query time. A critical correctness/compliance point.
- `[BP]` **Search = inverted index as a derived, eventually-consistent, permission-filtered read model** (18.7 + 20.7 + 20.3).

### 3.2 Recommendations / advice (18.5 / 19.2.7)

`[BP]`
- **What's recommended:** portfolio allocations, rebalancing suggestions, relevant products/insights.
- **Design (19.2.7):** the **offline/online split** — **offline** train models + precompute (using behavior + portfolio + market data — 20.5) → **online** serve fast via **two-stage retrieve→rank** — with **streaming freshness** (recent market moves — 20.5/9.6).
- `[BP]` Reuses the whole 19.2.7 shape — but constrained by regulation (§3.3).

### 3.3 The financial twist — suitability + explainability (15.8)

`[CS]` Financial advice is **regulated** — not just engagement optimization `[BP]`:
- **Suitability / fiduciary rules** (15.8): a recommendation must be **appropriate for the client's risk profile, goals, and constraints** — a **rules/compliance layer** filters/validates ML output (never recommend unsuitable products). This is a **hard business constraint**, not a soft ranking signal.
- **Explainability** (`[EMERGING]`): regulators + clients may require **why** a recommendation was made → prefer explainable models or provide rationale; **log the basis** (audit — 20.7). A pure black-box optimizer is risky in finance.
- **Auditability:** advice given is **recorded** (what, when, why, to whom — 20.7/15.8) for compliance.
- `[BP]` **Recs = 19.2.7 pipeline + a suitability/compliance filter + explainability + audit** — the financial constraints reshape a standard rec system.

### 3.4 AI services (off the critical path)

`[BP]`
- **Examples:** natural-language query ("show my tech holdings' performance"), document intelligence (parse statements/tax docs), insights/summaries, chatbots.
- **Design:** **separate services** (own scaling — 7.2), invoked **asynchronously or as non-blocking enrichment** — **never on the correctness-critical path** (money). Guard with **timeouts + fallbacks + circuit breakers** (11.3) — if the AI service is slow/down, the core platform still works (graceful degradation — 11.4).
- **Data:** AI consumes **derived data** (events, read models — 20.7), respecting **privacy/PII** (15.8) — mind what data feeds models (GDPR).
- `[BP]` **AI = additive, isolated, resilient enrichment** — enhances but never gates core financial operations.

### 3.5 Deep dives + bottlenecks

`[BP]`
- **Derived + eventually consistent** (20.7/10.5): search + recs lag the write side slightly — fine (they're advisory/read, not authoritative).
- **Permission-filtered results** (§3.1): search/recs must enforce authorization (20.3) — a compliance-critical correctness point.
- **Suitability/compliance filter** (§3.3): the key financial constraint on advice.
- **Isolation from the core** (§3.4): search/recs/AI failures must **degrade gracefully** (11.4) — the ledger + trading keep working.
- **Freshness vs cost** (19.2.7): recs balance batch + streaming; search index lag acceptable.
- **Bottleneck:** offline compute (recs — scale the batch/stream cluster) + search query fan-out (scatter-gather — 18.7); both scale horizontally, decoupled from the write side.
- `[BP]` **The lesson:** search/recs/AI = **derived, eventually-consistent, permission-filtered read-side services** — search is an **inverted-index read model built from events (18.7 + 20.7 + 20.3)**; recommendations reuse the **offline/online two-stage pipeline (19.2.7)** plus a **suitability/compliance filter + explainability + audit (15.8)**; AI is **isolated, non-critical-path, resilient enrichment (11.3/11.4)**. All decoupled from the CP ledger.

## 5. Real-World Analogy

Think of a **research-and-advice desk at a brokerage, working off the official records but never touching the vault**.

- **Search = a well-organized library catalog:** it lets you find instruments, past transactions, and documents quickly. The catalog is **built from the records** (and refreshes shortly after new records arrive), and — importantly — it **only shows you the shelves you're allowed to see** (your own transactions; an advisor sees only their clients').
- **Recommendations = a knowledgeable analyst's suggestions:** built by studying lots of data offline and tailored to you on the spot. But in a brokerage, the analyst **can't suggest anything unsuitable** for your risk profile — a **compliance officer vets the advice** (suitability), and the analyst must be able to **explain the reasoning**, which is **logged** for the record.
- **AI services = helpful assistants, not decision-makers:** a smart assistant can summarize your statements or answer "how did my tech stocks do?" — but if the assistant is out sick, **the brokerage still runs**: trades still settle, balances are still correct. The assistants **enhance** the experience; they never **gate** the money.
- **Never touch the vault:** the whole research desk reads from copies of the records; it never reaches into the official ledger to move money.

## 8–14. (Condensed)

**Advantages:** rich find/decide capabilities; derived read models scale independently (CQRS); recs reuse a proven pipeline; AI isolated + resilient; compliance built into advice.
**Disadvantages/cautions:** eventual consistency (search/recs lag — fine); suitability/explainability constraints on ML; permission-filtering complexity; AI data privacy (GDPR); operational surface (index + pipeline + models).
**When NOT to:** don't put search/recs/AI on the correctness-critical path; don't optimize advice for engagement over suitability; don't expose unfiltered (unauthorized) search results.
**Common mistakes:** search results leaking others' data (missing permission filter — compliance breach); black-box unexplainable advice; AI failures cascading into core flows (no isolation/breakers); recs ignoring suitability; treating the search index as a source of truth.
**Interview Qs:** 🟢 Why is search a derived read model? 🟡 How do you enforce authorization in search results? 🔴 How does financial advice differ from a consumer rec system (suitability/explainability)? How do you isolate AI? ⚫ Full design: search (inverted index/CQRS/permission-filtered), recs (19.2.7 + compliance), AI (isolated/resilient), all off the CP path.
**Production pitfalls:** permission-filter bugs (data leakage); stale index confusion; unsuitable recommendations (compliance); AI latency/outages leaking to core (no breaker); model data-privacy violations.
**Optimizations:** CDC-driven index updates (9.8); cache hot searches/recs (Part 6); precompute recs offline (19.2.7); scatter-gather search (18.7); async AI enrichment with fallbacks; explainable models + rationale logging.

## 15. Summary

Beyond core money movement, the platform must help users **find** things (search) and **decide** things (advice) — **derived, read-side, eventually-consistent** capabilities (20.7) that consume the event stream + market data and **never sit on the CP ledger path**. They reuse **18.7** (search) and **18.5/19.2.7** (recommendations), with a **financial twist**: advice is **regulated**. **Search** covers instruments, transactions, holdings, and documents via an **inverted index** (18.7 — text analysis + BM25 relevance + filters, sharded + scatter-gather), built as a **derived read model in CQRS style** — **fed from events via CDC/projections** (9.8/20.7), **eventually consistent**, and **rebuildable by replay** (20.7). Crucially, search results must be **permission-filtered** (a user searches only **their** transactions; an advisor only **their assigned clients'** — ReBAC — 20.3) — a compliance-critical correctness point (unfiltered results = data leakage). **Recommendations/advice** (allocations, rebalancing, insights) reuse the **19.2.7 pipeline** — the **offline/online split** (offline train + precompute from behavior + portfolio + market data → online **two-stage retrieve→rank**) with **streaming freshness** (recent market moves — 20.5) — but reshaped by the **financial twist**: a **suitability/fiduciary rules layer** (15.8) filters/validates ML output so recommendations are **appropriate** for the client's risk profile (a **hard business constraint**, not a soft signal), **explainability** is preferred/required (regulators + clients may demand *why* — log the rationale — `[EMERGING]`), and advice is **audited** (what/when/why/to whom — 20.7/15.8). **AI services** (natural-language query, document intelligence, insights, chatbots) are **separate services** invoked **asynchronously as non-blocking enrichment**, guarded by **timeouts + fallbacks + circuit breakers** (11.3) so failures **degrade gracefully** (11.4) — they **enhance but never gate** core financial operations — and they consume **derived data** with **privacy/PII care** (GDPR — 15.8). **Deep dives:** everything is **derived + eventually consistent** (advisory/read, not authoritative — fine), **permission-filtered** (compliance-critical), **suitability-constrained** (advice), and **isolated from the core** (graceful degradation). The **bottleneck** — offline rec compute + search query fan-out — scales horizontally, decoupled from the write side. In one line: **search (inverted-index derived read model, CQRS-fed, permission-filtered — 18.7/20.7/20.3) + recommendations (19.2.7 offline/online two-stage + suitability filter + explainability + audit — 15.8) + AI (isolated, non-critical-path, resilient enrichment — 11.3/11.4), all decoupled from the CP ledger**.

- **Q:** Recommendation pipeline? **A:** Reuse 19.2.7: offline train/precompute → online two-stage retrieve→rank + streaming freshness.
- **Q:** How does financial advice differ from consumer recs? **A:** Regulated — suitability/fiduciary filter (hard constraint) + explainability + audit; not engagement optimization.
- **Q:** Where do AI services sit? **A:** Off the correctness-critical path — async enrichment with timeouts/breakers/graceful degradation (11.3/11.4); never gate money.
- **Q:** Consistency of search/recs? **A:** Eventually consistent (advisory/read) — decoupled from the CP ledger.
- **Q:** AI + privacy? **A:** Consume derived data with PII/GDPR care (15.8).
