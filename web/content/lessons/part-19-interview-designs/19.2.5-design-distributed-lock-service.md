## 2. Problem statement

Design a **distributed lock service**: allow processes across many machines to **coordinate exclusive access** to a resource ("only one worker processes this job / only one leader acts at a time"). This is a **pure distributed-systems-correctness** design — the interesting part isn't scale, it's that **naive distributed locks are unsafe** (a holder can be presumed dead while still acting), requiring **leases + fencing + consensus** (8.3.6/8.3.8). It underlies leader election, job deduplication, and exclusive resource access.

## 3. The design (framework — 1.3.1)

### 3.1 Requirements

`[BP]`
- **Functional:** `acquire(resource) → lock (with owner + expiry)`, `release(resource)`, ideally `renew`; **at most one holder** at a time (mutual exclusion); support **many resources**.
- **Non-functional:** **safety first** (never two holders acting as if exclusive — the correctness bar); **liveness** (a crashed holder's lock is eventually released — no permanent deadlock); **availability** of the lock service; low-ish latency.
- `[BP]` **Key signal:** the hard requirement is **safety under failure** — a lock is only useful if it's **correct** when things go wrong (that's exactly when you need it). This is a **CP** service (10.7): prefer **refusing to grant** over granting incorrectly.

### 3.2 The fundamental problem (8.3.6)

`[CS]` Why naive locks fail `[BP]`:
- A process **acquires** a lock, then **pauses** (a long GC pause, a network partition, a slow disk) for longer than expected. The lock service (or others) **presume it dead** and the lock **expires / is granted to another process**. But the paused process **wakes up believing it still holds the lock** and acts on the resource → **two holders** → corruption/split-brain (8.3.6/8.1.3 — "is it dead?" is undecidable).
- **A lock with no expiry** risks **permanent deadlock** (holder crashes → lock never released). **A lock with an expiry** risks the **stale-holder** problem above. You can't escape this with timeouts alone.

### 3.3 The correct design — leases + fencing tokens (8.3.6)

`[CS]` The two mechanisms `[BP]`:
- **Lease (TTL):** the lock is granted for a **bounded time**; the holder must **renew** (heartbeat) to keep it. If the holder dies/pauses, the lease **expires** → liveness (no permanent deadlock). But leases alone don't fix the stale-holder problem (§3.2).
- **Fencing token:** each lock grant includes a **monotonically increasing token** (8.3.6). The holder **passes the token to the protected resource** on every operation; the resource **rejects any operation with a token lower than the highest it has seen**. So a **paused old holder** (with an old, lower token) that wakes up is **fenced out** — its writes are rejected — even though it thinks it holds the lock. `[BP]` **Fencing is what makes distributed locks actually safe** (8.3.6) — the resource, not the lock service, enforces exclusivity via the token.
- `[BP]` **Correct locking = lease (liveness) + fencing token (safety).** Both are required.

### 3.4 The backing store — consensus (8.3.8 / 8.3.3)

`[CS]` Where lock state lives `[BP]`:
- Lock state (who holds what, tokens) must be **consistent + fault-tolerant** → store it in a **consensus-backed coordination service**: **ZooKeeper** (ephemeral znodes + sequential nodes for locks/leader-election) or **etcd** (lease + compare-and-swap keys) — both built on **consensus** (ZAB/Raft — 8.3.3/8.3.8). The consensus layer gives a **single, agreed-upon** lock state despite failures — the source of monotonic fencing tokens too.
- This makes the lock service **CP** (10.7): during a partition, the minority side **can't grant locks** (it can't reach quorum — 8.3.4) — correctly refusing rather than risking two holders.
- `[BP]` **Consensus-backed store (etcd/ZooKeeper) = the safe foundation** (8.3.8); the fencing token is naturally the monotonic revision/zxid.

### 3.5 Redis/Redlock — the fast-but-debated alternative (8.3.6)

`[BP]`
- A common fast alternative: a **single-instance Redis lock** (`SET key val NX PX ttl`) or **Redlock** (across N Redis nodes). It's **fast + simple**, but **debated for correctness** (8.3.6): it relies on **timing/clock assumptions** (8.1.2) and, **without fencing tokens**, is vulnerable to the stale-holder problem. `[OPINION]` Fine for **efficiency locks** (an optimization — "avoid duplicate work, occasional overlap is OK") but **not for correctness locks** (where two holders would corrupt data) — for those, use consensus + fencing (8.3.6).
- `[BP]` **State the distinction: efficiency lock (fast, best-effort, Redis OK) vs correctness lock (must be safe → consensus + fencing).** This is the key senior insight.

### 3.6 Deep dives + bottlenecks

`[BP]`
- **Safety vs liveness** (§3.2/3.3): lease = liveness, fencing = safety; you need both.
- **Clock dependence** (8.1.2): TTL/leases assume bounded clock drift; fencing tokens make correctness **not** depend on clocks (the resource enforces monotonicity regardless of time) — a key reason fencing matters.
- **The lock service as a dependency:** it's on the critical path of coordinated operations → must be **HA** (consensus quorum — 8.3.4) and **fast**; a slow lock service slows everything.
- **CP behavior** (10.7): under partition, refuse to grant on the minority side — correctness over availability.
- **Granularity/scale:** many fine-grained locks; the consensus cluster handles metadata scale, but keep lock traffic reasonable (locks are coordination, not high-throughput data).
- **Bottleneck:** the consensus cluster's write throughput (all grants go through consensus) → keep locks coarse-enough + short-lived; it's a coordination service, not a data store.
- `[BP]` **The lesson (from 8.3.6/8.3.8):** a correct distributed lock = **lease (TTL, liveness) + fencing token (monotonic, safety) + consensus-backed store (etcd/ZooKeeper, CP)**. Redis/Redlock is fine for **efficiency** locks, not **correctness** locks. The lock service is CP and safety-first.

## 5. Real-World Analogy

Think of a **single key to a shared workshop**, handed out by a front desk.

- **The naive problem:** you take the key, then get stuck in a long phone call (a GC pause). The desk assumes you left, and hands the key to someone else. Now you finish your call, walk in **believing you still have the only key** — and two people are using the workshop at once.
- **Lease (TTL) = the key expires:** you must **check in periodically** to keep it. If you vanish, the key **automatically expires** so the workshop isn't locked forever (liveness).
- **Fencing token = a numbered badge the workshop's lock checks:** every time you use a machine, you swipe your **badge number**. The machine **remembers the highest badge it's seen** and **refuses any lower number**. So when you stroll in late with badge #6, but the machine has already served badge #7 (the new holder), your swipe is **rejected** — even though you think you're the rightful holder. The *machine* enforces exclusivity, not your belief.
- **Consensus desk = a fault-tolerant front desk:** the desk itself is replicated so it never gives contradictory answers; if half the building loses contact, that half **won't hand out keys** (it can't confirm with the others) — better to refuse than to double-issue.
- **Efficiency vs correctness lock:** if the key just avoids two people redundantly sweeping the floor, occasional overlap is harmless (a fast, casual key is fine). If it guards a machine that would **explode** with two operators, you need the badge-checking, fault-tolerant system.

## 7. Implementation Details

- **Lease (TTL) + renew** for liveness; **fencing token** (monotonic) enforced by the resource for safety (8.3.6) (§3.3).
- **Consensus-backed store** (etcd/ZooKeeper — 8.3.8, Raft — 8.3.3); fencing token = revision/zxid (§3.4).
- **CP behavior** (10.7): refuse to grant on the minority side under partition (§3.4).
- **Efficiency vs correctness lock** distinction (§3.5): Redis/Redlock for efficiency; consensus+fencing for correctness.
- Keep locks coarse-enough + short-lived; the lock service is coordination, not a data store (§3.6).

## 8–14. (Condensed)

**Advantages:** correct mutual exclusion under failure (lease+fencing+consensus); no permanent deadlock (lease); safe against stale holders (fencing); fault-tolerant (CP).
**Disadvantages/cautions:** consensus adds latency + is a critical dependency; requires the *resource* to check fencing tokens (not automatic); clock assumptions for leases; throughput-limited (coordination, not data).
**When NOT to use:** avoid distributed locks if you can design **lock-free** (idempotency/CAS/partitioning — 11.5/7.3) — they're a last resort; don't use Redis locks for correctness-critical exclusivity without fencing.
**Common mistakes:** lock with no expiry (permanent deadlock on crash); lease without fencing (stale-holder split-brain — §3.2); assuming a lock alone guarantees exclusivity (the resource must fence); Redlock for correctness locks; ignoring clock drift.
**Interview Qs:** 🟢 Why are distributed locks hard? 🟡 What do leases and fencing tokens each solve? 🔴 Why consensus-backed (CP)? Redis/Redlock — when OK, when not? ⚫ Full design: lease+fencing+consensus, safety-vs-liveness, clock dependence, efficiency-vs-correctness locks.
**Production pitfalls:** stale-holder split-brain (no fencing); GC-pause lock loss; clock skew breaking leases; lock service as a bottleneck/SPOF; deadlocks from ordering.
**Optimizations:** short leases + fast renew; coarse locks; lease caching; prefer lock-free designs (idempotency/CAS — 11.5) where possible; fence at the storage layer.

## 15. Summary

A **distributed lock service** lets processes across machines **coordinate exclusive access** to a resource, and it's a **pure correctness** design — the difficulty isn't scale but that **naive distributed locks are unsafe** (8.3.6). The **fundamental problem**: a lock holder can **pause** (long GC, partition, slow disk) past its assumed liveness; the service presumes it dead and grants the lock to another, but the paused process **wakes up still believing it holds the lock** and acts → **two holders → split-brain/corruption** (8.3.6/8.1.3 — "is it dead?" is undecidable). Timeouts alone can't escape this: **no expiry** → permanent deadlock on crash; **an expiry** → the stale-holder danger. The **correct design combines two mechanisms**: a **lease (TTL)** the holder must **renew** (gives **liveness** — a dead holder's lock expires), and a **fencing token** — a **monotonically increasing number** included with each grant that the holder passes to the **protected resource**, which **rejects any operation carrying a token lower than the highest it's seen** (gives **safety** — a paused old holder with a stale token is **fenced out** regardless of what it believes). **Fencing is what actually makes distributed locks safe**, because the *resource* enforces exclusivity, and correctness stops depending on clocks (8.1.2). Lock state lives in a **consensus-backed coordination service** — **etcd or ZooKeeper** (8.3.8, built on Raft/ZAB — 8.3.3) — giving a single agreed lock state and a natural monotonic token (revision/zxid); this makes the service **CP** (10.7): under partition the minority **refuses to grant** (can't reach quorum — 8.3.4) rather than risk two holders. The key senior distinction is **efficiency locks** (best-effort optimization to avoid duplicate work — a fast **Redis/Redlock** lock is acceptable) vs **correctness locks** (two holders would corrupt data — **require consensus + fencing**; Redlock is debated here because it leans on timing/clock assumptions and lacks fencing — 8.3.6, `[OPINION]`). **Deep dives:** safety-vs-liveness (fencing vs lease), clock dependence (fencing removes it), the lock service as an HA critical dependency, CP partition behavior, and the consensus cluster's write-throughput bottleneck (keep locks coarse + short-lived; it's coordination, not data). And always prefer **lock-free** designs (idempotency/CAS/partitioning — 11.5/7.3) when possible — distributed locks are a last resort. In one line: **lease (liveness) + fencing token (safety) + consensus-backed store (CP), efficiency-vs-correctness locks**.

## 16. Revision Notes (flashcard-ready)

- **Q:** Why are distributed locks hard? **A:** A holder can pause (GC/partition), be presumed dead, lose the lock, then wake up still acting → two holders/split-brain.
- **Q:** What does a lease (TTL) give? **A:** Liveness — a dead holder's lock expires (no permanent deadlock). But not safety alone.
- **Q:** What does a fencing token give? **A:** Safety — monotonic token; the resource rejects lower tokens, fencing out a stale old holder.
- **Q:** Correct lock = ? **A:** Lease (liveness) + fencing token (safety) + consensus-backed store (CP).
- **Q:** Backing store? **A:** etcd/ZooKeeper (8.3.8) on consensus (Raft/ZAB); token = revision/zxid.
- **Q:** CP behavior? **A:** Under partition, minority refuses to grant (no quorum) — correctness over availability.
- **Q:** Redis/Redlock? **A:** Fast; OK for efficiency locks; debated/unsafe for correctness locks (timing assumptions, no fencing).
- **Q:** Efficiency vs correctness lock? **A:** Efficiency = avoid duplicate work (overlap OK, Redis fine); correctness = two holders corrupt data (need consensus+fencing).
- **Q:** Who enforces exclusivity? **A:** The protected resource, by checking the fencing token — not the client's belief.
- **Q:** Better than a lock? **A:** Lock-free designs (idempotency/CAS/partitioning — 11.5/7.3) when possible.
