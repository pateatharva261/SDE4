## 2. Problem statement

Design a **rate limiter**: restrict how many requests a client may make in a time window (e.g., "100 requests/minute per user"), rejecting excess with **HTTP 429**. It's both a **reliability** control (protect capacity — 11.4) and a **security** control (throttle abuse — 15.7), and a classic interview design that hinges on the **algorithm** and the **distributed-state** problem. (This applies 15.7 — rate limiting concepts — as an end-to-end design; and 2.4.4 — the LLD.)

## 3. The design (framework — 1.3.1)

### 3.1 Requirements

`[BP]`
- **Functional:** limit requests per client per window; **reject excess** (429 + `Retry-After`); configurable rules (per-user/IP/API-key/endpoint — 15.7); ideally distributed (works across many instances).
- **Non-functional:** **low latency + low overhead** (it's on every request's hot path → must be cheap), **accurate** (enforce the limit, not wildly over/under), **highly available** (a broken limiter shouldn't break the API — fail-open vs fail-closed — §3.6), **scalable**.
- `[BP]` **Key signals:** it's **on every request** (must be fast + HA) and must enforce a **global** limit across a **distributed** fleet (§3.4 — the crux). Drive to **algorithm + distributed state**.

### 3.2 API / placement

`[BP]`
- Typically a **middleware/component at the API gateway** (12.6) or a **sidecar** (13.6/12.7 mesh) — enforce at the **edge**, rejecting excess **before** it consumes backend resources (11.4 load shedding — cheapest — 15.7).
- Interface: `allow(clientId, rule) → allowed | denied(retryAfter)`.
- `[BP]` **Enforce early** (gateway) + optionally per-service (defense in depth — 15.7).

### 3.3 Algorithm (15.7/2.4.4)

`[CS]` The core choice (15.7 §3.2) `[BP]`:
- **Token bucket (recommended default):** a bucket of N tokens, refilled at a rate; each request consumes one; empty → reject. **Allows bursts** (up to N) while capping the average → flexible, common. Needs **2 values per client**: token count + last-refill timestamp.
- **Leaky bucket:** fixed-rate drain (smooths output); good for steady downstream protection.
- **Fixed window:** count per window; simple but **boundary bursts** (2× at edges).
- **Sliding window (log or counter):** accurate, no boundary problem, but more memory/compute (sliding-window-counter is a cheap approximation).
- `[BP]` **Recommendation: token bucket** (burst-tolerant, cheap — 2 values/client) for general API rate limiting; **sliding-window-counter** if you need accuracy without the fixed-window boundary problem. (Detailed algorithm/object-model in 2.4.4.)

### 3.4 Distributed enforcement (the crux)

`[CS]` The hard part: enforce a **global** limit across **many instances** (7.1) `[CS]`:
- **The problem:** if each instance counts locally, a client hitting different instances can **exceed the intended global limit** (N instances → up to N× the limit — 15.7).
- **Solution: a shared counter store** — typically **Redis** (6.6 — fast, in-memory, atomic ops): all instances read/update the **same counter** per client → a **global** view.
- **Atomicity (critical):** the read-modify-write of the counter must be **atomic** to avoid race conditions (two requests both reading "99, allowed" then both incrementing to 101 — over-limit). Use **atomic Redis operations** (INCR, or a **Lua script** for token-bucket check-and-decrement atomically) → no races (11.5 CAS-like).
- **Performance:** the shared store adds a **network round-trip per request** (latency + a dependency on the hot path) → mitigate with: **atomic single-op** designs, **pipelining** (17.4), **local pre-check + async sync** (approximate — trade accuracy for speed), or **sharding** the counter store.
- `[BP]` **The distributed-state tradeoff:** a **strictly-global, atomic** shared counter (accurate but adds latency + a SPOF dependency) vs **local/approximate** limits (fast but less accurate). Discuss the tradeoff; the shared-Redis atomic approach is the common answer.

### 3.5 Data model

`[BP]`
- Per client: `{ clientId → tokens, lastRefill }` (token bucket) in Redis (6.6), with a **TTL** (expire idle clients → bound memory).
- Rules config: `{ rule → limit, window, dimension }`.
- `[BP]` Redis holds **soft state** (counters, TTL'd) — losing it means resetting limits (acceptable — fail-open — §3.6).

### 3.6 Deep dives + bottlenecks

`[BP]`
- **Atomicity / race conditions** (§3.4): atomic Redis ops / Lua script — the key correctness concern.
- **The counter store as a SPOF/bottleneck:** Redis is on **every request** → must be **HA** (6.6 — replication/cluster — 6.6) + **fast**; **shard** the counter store (7.3) for throughput; a slow/down Redis affects all requests → **fail-open** (allow on error — availability over strict limiting) vs **fail-closed** (deny — security over availability) — a deliberate choice (15.1/11.4). **Usually fail-open** for a rate limiter (don't break the API if the limiter is down) — unless it's a critical security control.
- **Dimensions** (15.7): per-user + per-IP + per-key + per-endpoint + global — combine (15.7); different rules per dimension.
- **Client behavior:** return **429 + `Retry-After`** so well-behaved clients **back off** (backpressure — 9.9); publish limits (headers: remaining/reset).
- **Multiple rules:** a request may match several rules (per-user AND per-IP) → check all, deny if any exceeded.
- **Accuracy vs performance** (§3.4): strict-atomic-global vs approximate-local.
- `[BP]` **The two hard parts: the algorithm (token bucket) + distributed atomic state (shared Redis).** The bottleneck is the **counter store** → HA + shard + atomic ops + fail-open.

## 5. Real-World Analogy

Think of a **club with multiple entrances** enforcing "each guest may enter at most 100 times an hour."

- **Token bucket = a stamp card refilled steadily:** each guest has an allowance (bucket of stamps) refilled over time; each entry uses a stamp; out of stamps → wait. A small **burst** (a few quick entries) is fine (bucket has slack), but the **sustained rate** is capped.
- **Distributed enforcement = doors sharing one guest ledger:** with **multiple doors** (instances), a sneaky guest could enter 100 times at **each** door if the doors count separately (exceeding the global 100). So all doors **check the same central ledger** (shared Redis) — a **global** count. Crucially, two doors checking "99, allowed" **simultaneously** must not both admit the guest (race) → the ledger update is **atomic** (one door at a time per guest — Redis atomic op/Lua).
- **The ledger is the bottleneck:** every entry consults the central ledger, so it must be **fast + always available + able to handle the load** (HA + sharded Redis). And if the ledger is **temporarily down**, do you **turn everyone away** (fail-closed) or **let everyone in** (fail-open)? Usually **let them in** — a broken counter shouldn't shut the club — unless the limit is a critical security barrier.
- **429 + Retry-After = "come back in 5 minutes":** a turned-away guest is told **when to return** so they don't keep hammering the door.

## 7. Implementation Details

- **Enforce at the gateway/edge** (12.6) as load shedding (11.4); defense-in-depth per-service (15.7).
- **Token bucket** (2 values/client) default; sliding-window-counter for accuracy (§3.3, 2.4.4).
- **Shared atomic counter in Redis** (6.6): atomic op / Lua script for check-and-decrement → **no races** (§3.4); TTL idle clients (§3.5).
- **HA + shard the counter store** (6.6/7.3); **fail-open** on error (usually — §3.6); pipeline/local-pre-check to cut latency (17.4).
- **Combine dimensions** (per-user/IP/key/endpoint/global — 15.7); check all matching rules.
- **429 + Retry-After + rate-limit headers** (§3.6, 9.9); layer with CAPTCHA/lockout for security abuse (15.7).

## 8–14. (Advantages / disadvantages / mistakes / questions / pitfalls / optimizations)

**Advantages:** simple, cheap, effective; dual reliability+security (15.7); global enforcement via shared atomic counter.
**Disadvantages/cautions:** counter store is a hot-path SPOF/bottleneck (HA + shard); atomicity required (races); accuracy vs latency tradeoff; fail-open/closed choice.
**Common mistakes:** local (per-instance) counting → exceeds global limit; non-atomic RMW → races/over-limit; per-IP-only (NAT/botnets — 15.7); no 429/Retry-After; fail-closed breaking the API.
**Interview Qs:** 🟢 What algorithm + why token bucket? 🟡 How to enforce globally across instances (shared atomic Redis)? Why atomic? 🔴 The counter store as a SPOF — HA/shard/fail-open? Accuracy vs latency? ⚫ Full distributed rate-limiter design + dimensions + graceful behavior + the shared-state tradeoff.
**Production pitfalls:** race-condition over-limit (non-atomic); Redis bottleneck/outage affecting all requests; NAT false positives (per-IP); latency from the counter round-trip.
**Optimizations:** atomic single-op/Lua; pipelining (17.4); sharded HA Redis; local pre-check + async sync (approximate); combined dimensions; 429 + backoff.

## 15. Summary

A **rate limiter** restricts requests per client per window, rejecting excess with **429**, serving as both a **reliability** (protect capacity — 11.4) and **security** (throttle abuse — 15.7) control — this design applies **15.7** (concepts) + **2.4.4** (LLD) end-to-end. Its **key signals**: it's **on every request** (must be **fast + low-overhead + HA**) and must enforce a **global** limit across a **distributed fleet** (7.1) — the crux. Enforce it **at the gateway/edge** (12.6, defense-in-depth per-service — 15.7) as early **load shedding** (11.4). The **algorithm** (15.7/2.4.4): **token bucket** (recommended — burst-tolerant, cheap: 2 values/client — token count + last-refill), or leaky bucket (steady rate), fixed window (simple, boundary bursts), or **sliding-window-counter** (accurate, cheap approximation). The **distributed-state problem** is the hard part: **local per-instance counting lets a client exceed the global limit** (N instances → up to N× — 15.7), so use a **shared counter store (Redis — 6.6)** for a **global** view — with **atomic** read-modify-write (atomic Redis ops / a **Lua script** for check-and-decrement) to prevent **race conditions** (two requests both reading "allowed" then both incrementing over-limit — 11.5 CAS-like). This adds a **round-trip per request** (latency + a hot-path dependency), traded off against accuracy (strict-atomic-global vs local/approximate). The **bottleneck/SPOF is the counter store**: Redis is on **every request** → must be **HA + fast + sharded** (6.6/7.3), and on failure you choose **fail-open** (allow — availability over strict limiting — usually preferred for a rate limiter, so a broken limiter doesn't break the API) vs **fail-closed** (deny — security over availability — for critical security limits) (15.1/11.4). **Deep dives:** **dimensions** (per-user/IP/key/endpoint/global — combine — 15.7), **client behavior** (**429 + `Retry-After`** + rate-limit headers so well-behaved clients back off — 9.9), **multiple rules** (check all matching, deny if any exceeded), and **accuracy vs performance**. The **two hard parts: the algorithm (token bucket) + distributed atomic state (shared Redis)** — with the counter store as the bottleneck (HA + shard + atomic ops + fail-open). A clean, high-leverage design reusing 15.7/2.4.4/6.6.
