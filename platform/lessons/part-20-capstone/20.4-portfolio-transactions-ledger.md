## 1. Learning Objectives

After this lesson you will be able to:

- Design the **Portfolio** and **Ledger/Transactions** contexts (20.1) — the **financial heart** of the platform.
- Apply the **double-entry ledger** (18.3/19.2.3) as the immutable, ACID, auditable source of truth for money + holdings.
- Guarantee **exactly-once effects** for trades/transfers via **idempotency** (11.5) — no double-execution.
- Separate the **write model** (the ledger — strongly consistent, CP — 10.7) from the **read model** (portfolio views — cached, eventually consistent) — a CQRS setup (20.7).
- Reason about **consistency boundaries**: what must be transactional together vs what can be eventual.

## 2. Motivation

This is where **money lives** — the lesson the whole platform's correctness-first priority (20.1/20.2) was built for. A wrong balance, a lost trade, or a double-executed transfer is unacceptable and possibly illegal. So the Portfolio/Ledger contexts apply the **payment-system discipline** (19.2.3) at platform scale: **idempotency, an immutable double-entry ledger, ACID, CP, and audit**.

## 3. The design (framework — 1.3.1)

### 3.1 Portfolio vs Ledger (two models)

`[BP]`
- **Ledger (write model — source of truth):** the **immutable, append-only, double-entry** record of every financial event (trades, transfers, fees, dividends). It is **the** authority on money + positions (18.3/19.2.3).
- **Portfolio (read model — derived):** the **current view** clients/advisors see — positions, balances, valuations, performance. **Derived from the ledger** (+ market data for valuation — 20.5), **cached** and **eventually consistent** (10.5) — a **materialized read model** (CQRS — 20.7).
- `[BP]` **Split write (CP ledger) from read (cached, eventual portfolio)** — the central structural decision (CQRS — 20.7). The books are exact; the dashboard is a fast derived projection.

### 3.2 The double-entry ledger (18.3/19.2.3)

`[CS]` The financial source of truth `[BP]`:
- Every transaction = **balanced debit + credit** entries (money/assets move *between* accounts; totals sum to zero → self-checking — 18.3). Buying a stock: debit cash, credit the security position (+ a fee entry). This makes the system **auditable + verifiable** (the books must balance).
- **Immutable / append-only:** never update or delete an entry; **corrections are new compensating entries** (event-sourcing flavor — 20.7). Gives a **complete, tamper-evident audit trail** (15.8/SOX).
- **ACID + strongly consistent** (5.2.1): a transaction's entries commit atomically; balances are correct at commit. Store in a **strongly-consistent DB** — a relational or **globally-distributed SQL** store (18.3 — Spanner/Cockroach lineage) for ACID at scale. **CP** (10.7): prefer refusing over risking an incorrect balance.
- `[BP]` **The ledger is sacred: immutable, balanced, ACID, CP, audited** — exactly as in 19.2.3, now the platform's core.

### 3.3 Executing transactions — idempotency + exactly-once effects (11.5)

`[BP]`
- Every money-moving operation (trade, transfer, payment) carries an **idempotency key** (11.5): a **retry** (client timeout, network glitch — 8.1.1) **returns the original result, never re-executes** → **exactly-once effects** (at-least-once + idempotency — 9.4). No double-buys, no double-charges.
- **A trade is a workflow**, often spanning services (validate → reserve funds → place order → settle → post to ledger → update portfolio) that **can't be one ACID transaction** across services → a **saga** (20.6/11.7) with compensations, and a **transactional outbox** (Part 9/12.5) to atomically post-and-emit — **no dual writes**.
- `[BP]` **Idempotency + saga + outbox** make trades safe and recoverable (details in 20.6).

### 3.4 Consistency boundaries (the design judgment)

`[CS]` What must be **transactionally together** vs **eventual** `[BP]`:
- **Transactional (strong, same context):** the double-entry ledger entries of a single transaction (debits + credits must commit together — an invariant — 12.4). Account balances that back a **hard invariant** (can't overdraw) need strong consistency / a real transaction (or a saga with reservation + compensation across contexts).
- **Eventual (derived, cross-context):** the **portfolio read view**, valuations (depend on market data — 20.5), reporting (20.7), search indexes (20.8), notifications (19.1.4) — all **derived** and **eventually consistent** (10.5), updated from **ledger events** (Part 9/CDC — 9.8).
- `[BP]` **Keep invariant-bound data together (strong); let derived data be eventual** (12.4/18.6). This is the core consistency judgment — over-doing strong consistency kills scale; under-doing it corrupts money.

### 3.5 HLD

`[BP]`
- **Transaction/trade service** (idempotent — §3.3) → orchestrates the workflow (saga — 20.6) → **posts to the ledger** (ACID, immutable — §3.2) → emits **ledger events** (outbox — Part 9).
- **Ledger store:** strongly-consistent distributed SQL (18.3); append-only; partitioned by account (7.3) keeping per-account transactions ACID.
- **Portfolio read model:** consumes ledger events + market data → maintains **materialized positions/balances/valuations** in a fast read store + cache (Part 6/20.9); serves the read-heavy dashboard (20.2).
- **Reporting/audit** (20.7): the immutable ledger + event stream = statements, tax, regulatory reports, and the audit trail.
- `[BP]` **The lesson:** portfolio/ledger = **immutable double-entry ACID CP ledger (write, source of truth) + idempotent exactly-once transactions (saga + outbox) + a derived, cached, eventually-consistent portfolio read model (CQRS)**, with **invariant-bound data kept strongly consistent** and **derived data eventual**. The payment discipline (19.2.3) at platform scale.

## 5. Real-World Analogy

Think of a **brokerage's official books versus the account statement you see online**.

- **The ledger = the official books, in ink, balanced, audited:** every trade is recorded as **matching debits and credits** (cash out, shares in) that must balance to the penny. Nothing is ever erased — a mistake becomes a **new correcting entry**. Auditors can reconstruct everything. This is the **truth**.
- **Your online portfolio = a printed statement generated from those books:** it's fast and convenient, but it's a **snapshot derived from the ledger** (plus current prices). If it lags the official books by a moment, that's fine — it's a *view*, not the source of truth.
- **Idempotency key = the order confirmation number:** if your "buy" click times out and you retry, the broker sees the same confirmation number and **doesn't buy twice** — you get the original fill.
- **Consistency boundaries = what must balance instantly vs what can catch up:** the debit and credit of a single trade must post **together** (or not at all) — that's sacred. But your **performance chart, tax report, and search index** can update **a little later** — they're derived and can catch up.
- **Why not make the fast statement the source of truth?** Because a fast, mutable statement can't be audited or trusted with money — the slow, immutable, balanced books must reign.

## 7. Implementation Details

- **Ledger:** immutable double-entry, ACID, CP, strongly-consistent distributed SQL (18.3), partitioned by account (7.3) (§3.2).
- **Transactions:** idempotency keys → exactly-once effects (11.5); workflow as a saga (20.6/11.7) + transactional outbox (Part 9/12.5) (§3.3).
- **Portfolio read model:** materialized positions/balances/valuations from ledger events + market data (20.5), cached (Part 6/20.9), eventually consistent (10.5) — CQRS (20.7) (§3.1/3.5).
- **Consistency boundaries:** invariant-bound (ledger entries, hard balances) strong; derived (portfolio/reporting/search/notifications) eventual (12.4/18.6) (§3.4).

## 8–14. (Condensed)

**Advantages:** correctness + auditability (immutable balanced ACID ledger); safe retries (idempotency); fast read dashboard (cached derived model); clean scale (strong only where needed, eventual elsewhere).
**Disadvantages/cautions:** CQRS adds read/write divergence lag (eventual reads); ledger's strong consistency caps write throughput (accepted); saga complexity for cross-service trades (20.6); derived views must be rebuildable.
**When NOT to:** don't make the read model the source of truth; don't use eventual consistency for ledger entries/hard invariants; don't skip idempotency on money moves.
**Common mistakes:** mutable/updatable balances (no audit); reading balances off a stale cache to authorize a trade (must check the ledger); dual writes (use outbox); one giant transaction across contexts (use sagas); over-applying strong consistency everywhere (kills scale).
**Interview Qs:** 🟢 Why a double-entry immutable ledger? 🟡 How do you prevent double-execution of a trade (idempotency)? 🔴 Write model vs read model (CQRS) — what's strong vs eventual? ⚫ Full portfolio/ledger design: ledger, idempotent saga+outbox transactions, derived read model, consistency boundaries.
**Production pitfalls:** ledger imbalance from partial failures (reconcile — 19.2.3); stale read model misinforming decisions; authorizing off cached balances; saga compensation gaps; hot account partitions (7.4).
**Optimizations:** partition ledger by account (7.3) keeping per-account ACID; materialized read views + cache (Part 6); event-driven read-model updates (CDC — 9.8); reconciliation jobs (19.2.3); snapshot balances for fast valuation.

## 15. Summary

The **Portfolio** and **Ledger/Transactions** contexts are the **financial heart** of the platform — the reason the whole correctness-first priority (20.1/20.2) exists — and they apply the **payment-system discipline (19.2.3) at platform scale**. The structure splits **two models**: the **ledger (write model, source of truth)** is the **immutable, append-only, double-entry** record of every financial event, and the **portfolio (read model)** is the **current view** clients see (positions, balances, valuations), **derived from the ledger + market data**, **cached and eventually consistent** — a **CQRS** split (20.7). The **double-entry ledger** (18.3) records each transaction as **balanced debit + credit** entries (money/assets move between accounts, totals sum to zero → **self-checking**), is **immutable/append-only** (corrections are new compensating entries → tamper-evident audit trail for SOX — 15.8), and is **ACID + strongly consistent + CP** (10.7) stored in a **globally-distributed SQL** engine (18.3) for ACID at scale — **sacred: immutable, balanced, ACID, CP, audited**. Every money-moving operation carries an **idempotency key** so retries **return the original result rather than re-executing** → **exactly-once effects** (11.5/9.4 — no double-buys); and because a trade is a **multi-service workflow** that can't be one ACID transaction, it runs as a **saga** (20.6/11.7) with compensations plus a **transactional outbox** (Part 9/12.5) to atomically post-and-emit (**no dual writes**). The key **consistency judgment**: keep **invariant-bound data transactionally together and strongly consistent** (a transaction's ledger entries must commit atomically; hard balances that can't overdraw), but let **derived, cross-context data be eventual** (the portfolio read view, valuations from market data, reporting, search indexes, notifications — all updated from **ledger events via CDC/outbox** — 9.8) — over-applying strong consistency kills scale, under-applying it corrupts money (12.4/18.6). The **HLD**: an idempotent transaction service orchestrates the saga and **posts to the ACID immutable ledger** (partitioned by account — 7.3, keeping per-account transactions ACID) emitting **ledger events**; a **portfolio read model** consumes those events + market data into **materialized, cached, eventually-consistent** positions/balances/valuations serving the read-heavy dashboard (20.2); and reporting/audit (20.7) derive from the same immutable ledger + event stream. In one line: **immutable double-entry ACID CP ledger (write) + idempotent exactly-once transactions (saga + outbox) + derived cached eventual portfolio read model (CQRS), with invariant-bound data strong and derived data eventual**.

## 16. Revision Notes (flashcard-ready)

- **Q:** Write model vs read model? **A:** Ledger = immutable ACID CP source of truth (write); Portfolio = derived, cached, eventually-consistent view (read) — CQRS.
- **Q:** Ledger properties? **A:** Double-entry balanced, immutable/append-only, ACID, strongly consistent, CP, auditable (18.3).
- **Q:** How correct a mistake in an immutable ledger? **A:** New compensating entry — never update/delete.
- **Q:** Prevent double-execution of a trade? **A:** Idempotency key → return original result on retry (exactly-once effects — 11.5).
- **Q:** Why is a trade a saga? **A:** Multi-service workflow (validate→reserve→execute→settle→post) can't be one ACID transaction → saga + compensations + outbox (20.6).
- **Q:** What's strong vs eventual? **A:** Strong: ledger entries + hard-invariant balances. Eventual: portfolio view, valuations, reporting, search, notifications (derived).
- **Q:** How is the portfolio view updated? **A:** From ledger events (CDC/outbox — 9.8) + market data → materialized, cached.
- **Q:** Can you authorize a trade off a cached balance? **A:** No — check the authoritative ledger; caches are for display, not authorization.
- **Q:** Ledger scaling? **A:** Partition by account (7.3) keeping per-account transactions ACID; reconcile (19.2.3).
