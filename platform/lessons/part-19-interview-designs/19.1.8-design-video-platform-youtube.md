## 2. Problem statement

Design a **video platform** (YouTube-style): users **upload videos**, which are **processed** and made available for **streaming** to viewers worldwide, with **metadata** (title/description), **search** (18.7), **view counts / likes**, and **recommendations** (18.5). It composes the **streaming architecture** (18.5) with an **upload + transcoding pipeline** — the write side that 18.5 (delivery-focused) didn't detail.

## 3. The design (framework — 1.3.1)

### 3.1 Requirements

`[BP]`
- **Functional:** upload video; **transcode** (multiple resolutions/formats); **stream** (adaptive bitrate — 18.5); metadata (title/desc/tags); **search** (18.7); **view counts + likes**; **recommendations** (18.5).
- **Non-functional:** **massive storage + bandwidth** (video is huge), **low-latency global streaming** (CDN — 18.4), **read-heavy** (views ≫ uploads), **highly available**, **scalable upload/transcoding**.
- `[BP]` **Key signals:** **video = huge + bandwidth-heavy → CDN + object storage** (18.4/4.1.3); **uploads need async transcoding** (9.1); **read-heavy streaming** (18.5). → compose 18.5 (delivery) + a transcoding pipeline.

### 3.2 The upload + transcoding pipeline (the write side)

`[CS]` Uploading a video triggers an **async processing pipeline** (9.1) `[CS]`:
1. **Upload:** client uploads the raw video (chunked/resumable — like 19.1.7) to **object storage** (4.1.3).
2. **Enqueue a transcoding job** (9.1): the upload triggers a job → don't block the user (async — the video is "processing").
3. **Transcoding (the heavy step):** transcode the raw video into **multiple resolutions + bitrates** (240p…4K) and **formats/segments** for **adaptive bitrate streaming** (18.5 — HLS/DASH: segmented, multi-bitrate) → CPU-intensive, **parallelized** across a fleet of workers (17.3 — CPU-bound → parallelism), often **per-segment** (split → transcode segments in parallel → assemble).
4. **Store outputs** (4.1.3) + **update metadata** (video ready, available renditions).
5. **Distribute to CDN** (18.4/18.5): **pre-position** popular content at the edge; content served from the CDN, not the origin.
- `[BP]` **The upload/transcoding pipeline is async + parallel** (9.1/17.3): upload to object storage → enqueue → transcode to multi-bitrate segments (CPU-heavy, parallel) → store + CDN. This is the write side 18.5 (which focused on delivery) implies.

### 3.3 Delivery — CDN + adaptive bitrate (18.5/18.4)

`[BP]` (See 18.5 for full detail):
- Video **streamed from the CDN edge** (18.4 — pre-positioned near viewers) via **adaptive bitrate** (18.5 — player picks bitrate for its bandwidth, segment by segment).
- **Separate video (CDN data plane) from metadata (services)** (18.5) — bytes bypass the app.
- `[BP]` Delivery = **18.5's video subsystem** (CDN + ABR + pre-positioning).

### 3.4 Metadata + view counts

`[BP]`
- **Metadata:** video info (title/desc/tags/renditions/uploader) in a **DB** (5.x) — read-heavy → cache (Part 6); **search** via an inverted index (18.7).
- **View counts (high-write):** every view increments a counter → **millions of increments** → **don't do a synchronous DB write per view** (like 19.1.1 analytics) → **emit view events** (async — 9.1/18.1) → **aggregate** (batch/stream — 9.6/9.7) → periodically update the displayed count (eventually consistent — 10.5). Same for likes.
- `[BP]` **View counts are a high-write aggregation** → async events + aggregation (18.1/9.6), not per-view DB writes. Metadata is read-heavy → cached (Part 6).

### 3.5 Recommendations + search

`[BP]`
- **Recommendations** (18.5): watch/interaction events → log → batch/stream + ML → cached recs (18.5 §3.5).
- **Search** (18.7): metadata + captions indexed in a search engine (inverted index — 18.7).
- `[BP]` Reuse 18.5 (recs) + 18.7 (search).

### 3.6 Deep dives + bottlenecks

`[BP]`
- **Transcoding scale** (§3.2, 17.3): CPU-bound → large parallel worker fleet; per-segment parallelism; autoscale (13.5) for upload spikes.
- **Storage/bandwidth cost** (17.6): video is huge → **CDN offload** (18.4) + tiered storage (16.2/4.1.3 — hot on fast, cold on cheap); this is a major **cost** driver (17.6).
- **View-count accuracy vs cost** (§3.4): async aggregation (eventually consistent — 10.5); approximate for popular videos.
- **Global delivery** (18.4/13.8): CDN + multi-region.
- **Live streaming** (extension): lower-latency pipeline (segment + distribute in near-real-time).
- **Bottleneck:** transcoding (parallel fleet) + bandwidth/storage cost (CDN + tiered storage). Delivery scales via CDN (18.4); metadata via cache + sharding.
- `[BP]` **The lesson:** a video platform = **18.5's streaming architecture (CDN + ABR delivery, recs) + an async parallel transcoding pipeline (9.1/17.3) + object storage (4.1.3) + async view-count aggregation (18.1) + search (18.7)**. The write side (transcoding) + delivery (CDN/ABR) + cost are the interesting parts.

## 5. Real-World Analogy

Think of a **TV production + broadcast operation**: producing content once, then broadcasting it to millions.

- **Upload + transcoding = producing broadcast-ready versions:** a creator submits a **raw master tape** (upload to object storage). The studio then **produces multiple versions** — HD, SD, mobile-quality (transcoding to multi-bitrate) — a **slow, compute-heavy job** done by a **large team working in parallel** (parallel transcoding fleet), often by **splitting the tape into reels and processing them simultaneously** (per-segment parallelism). This happens **after** the creator submits (async — they don't wait); the video shows "processing."
- **Delivery = broadcasting from local stations:** the finished versions are **shipped to thousands of local broadcast stations** (CDN — pre-positioned) so viewers watch from a **nearby station**, and each viewer's device **picks the quality matching their connection** (adaptive bitrate). The **actual video signal doesn't route through headquarters** (video/CDN separate from the control services).
- **View counts = tallying viewership without stopping the broadcast:** counting **millions of views** isn't done by **updating a master ledger on every single view** (that would bury the accountants) — instead, stations **report viewership periodically** and it's **tallied up** (async aggregation), showing an **approximate, slightly-delayed count**.
- **Cost = the dominant concern:** producing and broadcasting **video is enormously expensive** (storage + bandwidth), so you **keep popular content at local stations** (CDN offload) and **archive rarely-watched content to cheap deep storage** (tiered storage).

## 7. Implementation Details

- **Upload → object storage** (4.1.3, chunked/resumable — like 19.1.7) → **enqueue transcoding job** (9.1, async).
- **Transcode** (§3.2, 17.3): multi-bitrate segments for ABR (18.5); **CPU-bound → parallel worker fleet** (per-segment); autoscale (13.5) for upload spikes.
- **Delivery** (§3.3, 18.4/18.5): CDN + pre-positioning + ABR; **separate video (CDN) from metadata (services)**.
- **Metadata** (5.x) — read-heavy → cache (Part 6); **search** via inverted index (18.7).
- **View counts/likes** (§3.4): **async view events** (18.1/9.1) → **aggregate** (9.6) → eventually-consistent counts (10.5); don't write per view.
- **Recommendations** (18.5): events → log → batch/stream + ML → cached recs.
- **Cost** (17.6): CDN offload + tiered storage (16.2/4.1.3) — video is the cost driver.

## 8–14. (Advantages / disadvantages / mistakes / questions / pitfalls / optimizations)

**Advantages:** scalable global delivery (CDN + ABR); async parallel transcoding; read-heavy metadata cached; view counts scale via async aggregation; cost-managed (CDN offload + tiered storage).
**Disadvantages/cautions:** transcoding is compute-heavy (cost — 17.6); video storage/bandwidth cost dominates (17.6); eventually-consistent counts; complex pipeline.
**Common mistakes:** synchronous transcoding (blocking upload); streaming video through app services (should be CDN — 18.5); per-view DB writes (should be async aggregation); no ABR (buffering); ignoring cost (huge bandwidth/storage bills — 17.6).
**Interview Qs:** 🟢 Why async transcoding + CDN delivery? 🟡 The transcoding pipeline (parallel, per-segment, multi-bitrate)? View counts at scale? 🔴 CDN/ABR delivery + cost management + separate video/metadata? ⚫ Full design = 18.5 streaming + transcoding pipeline + view aggregation + search/recs.
**Production pitfalls:** transcoding backlog under upload spikes (autoscale — 13.5); bandwidth/storage cost blowout (CDN + tiered storage — 17.6); buffering (no ABR); per-view write overload; video served from origin (not CDN).
**Optimizations:** async parallel per-segment transcoding; CDN + pre-positioning + ABR (18.5); async view-count aggregation (18.1); tiered storage + CDN offload (cost — 17.6); metadata cache (Part 6); autoscaling transcoding (13.5).

## 15. Summary

A **video platform** (YouTube) lets users **upload videos** that are **processed and streamed** to viewers worldwide, with metadata, search (18.7), view counts, and recommendations (18.5) — it **composes the streaming architecture** (18.5) with an **upload + transcoding pipeline** (the write side 18.5's delivery-focus omitted). **Key signals:** **video is huge + bandwidth-heavy → CDN + object storage** (18.4/4.1.3); **uploads need async processing** (9.1); **streaming is read-heavy** (18.5). The **upload + transcoding pipeline** (async — 9.1): the client uploads the raw video (chunked/resumable — like 19.1.7) to **object storage** (4.1.3), which **enqueues a transcoding job** (async — the user doesn't wait; the video is "processing"); **transcoding** (the heavy, CPU-bound step — 17.3) produces **multiple resolutions + bitrates** and **segmented formats for adaptive bitrate streaming** (18.5 — HLS/DASH), **parallelized** across a worker fleet (17.3 — often **per-segment**: split → transcode in parallel → assemble); outputs are **stored** (4.1.3) + metadata updated (ready) + **distributed to the CDN** (18.4/18.5 — pre-positioned near viewers). **Delivery** is **18.5's video subsystem**: video **streamed from the CDN edge** via **adaptive bitrate** (player picks bitrate for its bandwidth), with **video (CDN data plane) separate from metadata (services)** (bytes bypass the app). **Metadata** (title/desc/renditions) is read-heavy → **cached** (Part 6) + **searchable** via an inverted index (18.7). **View counts** are a **high-write aggregation** — millions of increments → **not a synchronous DB write per view** (like 19.1.1) → **emit view events** (async — 18.1/9.1) → **aggregate** (batch/stream — 9.6) → periodically update the displayed (eventually-consistent — 10.5) count. **Recommendations** reuse 18.5's ML pipeline. **Deep dives**: **transcoding scale** (parallel fleet + autoscaling for spikes — 13.5), **storage/bandwidth cost** (the dominant driver — 17.6 → **CDN offload** + **tiered storage** — hot on fast, cold/rarely-watched on cheap — 16.2/4.1.3), **view-count accuracy vs cost** (async/approximate — 10.5), and **global delivery** (CDN + multi-region — 13.8). The **bottleneck** is **transcoding** (parallel fleet) + **bandwidth/storage cost** (CDN + tiered storage); delivery scales via CDN, metadata via cache + sharding. A video platform = **18.5's streaming architecture (CDN + ABR delivery + recs) + an async parallel transcoding pipeline (9.1/17.3) + object storage (4.1.3) + async view-count aggregation (18.1) + search (18.7)** — with transcoding (write), CDN/ABR (delivery), and cost as the interesting parts.
