## 1. Learning Objectives

After this lesson you will be able to:

- Design **autocomplete/typeahead**: given a **prefix**, return the **top-K most relevant completions** with **very low latency** (every keystroke).
- Use the right data structure — a **trie (prefix tree)** with **precomputed top-K per node**, and/or an inverted-index/prefix approach (18.7).
- Explain the **precompute + cache** strategy — autocomplete is **latency-critical** (17.2) → precompute top-K, cache aggressively (Part 6).
- Handle **ranking** (popularity/frequency, personalization), **freshness** (update suggestions from query logs — 9.6), and **scale** (sharding — 7.3).
- Recognize it as a **precomputed-top-K + low-latency-cache** design.

---

## 2. Problem statement

Design **search autocomplete** (typeahead): as a user **types a prefix**, instantly suggest the **top-K most likely completions** (e.g., "sys" → "system design," "systematic," ...). It fires on **every keystroke** → **extremely latency-sensitive** (17.2 — must feel instant, single-digit-to-low-ms), **read-heavy**, and **top-K-ranked** — a **precompute + cache** design distinct from full search (18.7).

---

## 3. The design (framework — 1.3.1)

### 3.1 Requirements

`[BP]`
- **Functional:** given a **prefix**, return **top-K completions** (ranked by relevance/popularity); update suggestions as popularity changes.
- **Non-functional:** **very low latency** (fires per keystroke → must be **~ms** — 17.2), **read-heavy** (queries ≫ updates), **scalable**, **fresh-ish** (trending queries surface — eventual is fine — 10.5).
- `[BP]` **Key signal: latency-critical + read-heavy + top-K per prefix** → **precompute top-K + cache aggressively** (Part 6). This is *not* full-text search (18.7) — it's prefix → precomputed top-K.

### 3.2 The data structure — trie with precomputed top-K

`[CS]` The core structure `[CS]`:
- **Trie (prefix tree):** a tree where each **node = a prefix**, edges = characters → walking the prefix leads to the node; the subtree under it holds all completions of that prefix. Natural for prefix queries.
- **Precomputed top-K at each node:** store the **top-K completions (by rank) directly at each prefix node** → a query is: **walk to the prefix node → return its precomputed top-K** — **O(prefix length)** + a constant → **extremely fast** (no ranking at query time).
- **Alternative:** precomputed **prefix → top-K** in a **KV store / cache** (Redis — 6.6) — same idea (precompute, look up), simpler to shard.
- `[BP]` **The key: precompute the top-K per prefix** (offline) so the query is a **lookup, not a computation** — because it's latency-critical (17.2). Ranking happens **offline** (§3.4), not per keystroke.

### 3.3 Precompute + cache (latency-critical)

`[CS]` Autocomplete is **latency-critical** → **precompute + cache** `[BP]`:
- **Precompute offline:** periodically (batch — 9.7) compute the **top-K per prefix** from query logs (§3.4) → build the trie / prefix→top-K map.
- **Cache aggressively** (Part 6/6.6): the prefix→top-K results are **highly cacheable** (read-heavy, popular prefixes) → serve from **in-memory cache** (Redis/CDN — 6.6/18.4) → **~ms latency**. Popular prefixes → near-100% cache hit.
- **Edge/CDN** (18.4): can even serve suggestions from the edge for global low latency.
- `[BP]` **The strategy: precompute top-K offline + serve from cache** → queries are cache lookups (fast). Do **zero ranking at query time** — it's all precomputed. Latency (17.2) drives everything.

### 3.4 Ranking + freshness (from query logs)

`[CS]` How are the top-K chosen + kept fresh? `[BP]`:
- **Ranking signals:** **query frequency/popularity** (how often each completion is searched), recency/trending, personalization, context.
- **From query logs** (9.6): user queries are **logged as events** (18.1) → **stream/batch processing** (9.6/9.7) **aggregates query frequencies** → periodically **recompute top-K per prefix** → update the trie/cache. **Trending** queries surface via recent-window aggregation (9.6 windowing).
- `[BP]` **Freshness = periodic recompute from query-log aggregation** (9.6/18.1). Autocomplete **learns from what people search** — a data pipeline (like recommendations — 18.5) feeding the precomputed top-K. Updates are **async/eventual** (10.5 — fine).

### 3.5 Scale (sharding)

`[BP]`
- **Sharding** (7.3): the trie / prefix→top-K map is huge → **shard by prefix** (e.g., first characters) across nodes; queries route to the right shard.
- **Read scaling** (Part 6/7.5): cache + replicas handle the read-heavy load.
- `[BP]` Shard by prefix (7.3); cache-heavy reads (Part 6).

### 3.6 Deep dives + bottlenecks

`[BP]`
- **Latency** (17.2): the whole design is about **ms latency** → precompute + cache + no query-time ranking; watch tail latency (17.2).
- **Trie memory:** a full trie of all queries is large → **prune** (only keep prefixes/completions above a popularity threshold), compress (radix tree), or use prefix→top-K in a KV store.
- **Freshness vs cost** (§3.4): recompute frequency (trending needs frequent updates); real-time updates via streaming (9.6) for trending, batch for the bulk.
- **Personalization:** blend global top-K with user-specific (adds complexity — often global + light personalization).
- **Correctness:** handle typos (fuzzy — like search 18.7), multiple words.
- **Bottleneck:** none extreme — precompute + cache dissolves the latency concern; trie memory + freshness pipeline are the interesting parts.
- `[BP]` **The lesson:** autocomplete = **precomputed top-K per prefix (trie / prefix→top-K map) + aggressive caching** (Part 6) for **ms latency** (17.2), with **ranking + freshness from query-log aggregation** (9.6/18.1) and **sharding by prefix** (7.3). Precompute + cache is the whole game; it's distinct from full search (18.7).

## 5. Real-World Analogy

Think of a **shop assistant who finishes your sentence instantly** because they've memorized the most common requests for every opening phrase.

- **Precomputed top-K = memorized popular completions per prefix:** rather than **thinking hard** about what you might want each time you start speaking (ranking at query time — too slow), the assistant has **already memorized**, for every possible opening ("I'd like a sys..."), the **top few most common things people ask for** (precomputed top-K per prefix). When you start typing, they **instantly recall** the memorized list — no thinking required (a lookup, not a computation).
- **Cache = the most-asked phrases on the tip of the tongue:** the **most common openings** are so frequently used they're **instantly at hand** (cache) — the assistant almost never has to look anything up.
- **Learning from logs = updating the memorized lists from what people actually ask:** periodically, the assistant **reviews what everyone's been asking** (query logs) and **updates the memorized completions** — so **trending requests** rise to the top (freshness from aggregation), and stale ones fade.
- **Why it must be instant:** the assistant finishes your sentence **as you type each letter** — so there's **no time to think**; everything must be **pre-memorized and instantly recalled** (latency-critical → precompute + cache).

## 7. Implementation Details

- **Data structure** (§3.2): **trie with precomputed top-K per node** (or prefix→top-K in a KV store — 6.6); query = walk-to-prefix → return top-K (fast).
- **Precompute + cache** (§3.3, Part 6): precompute top-K offline; **cache aggressively** (Redis/CDN — 6.6/18.4) for ms latency (17.2); **no ranking at query time**.
- **Ranking + freshness** (§3.4, 9.6/18.1): aggregate query frequencies from logs (stream/batch); recompute top-K periodically; trending via recent-window aggregation (9.6); async/eventual (10.5).
- **Scale** (§3.5): shard by prefix (7.3); cache + replicas for reads (7.5/Part 6).
- **Memory** (§3.6): prune low-popularity prefixes/completions; compress (radix tree).
- **Correctness:** fuzzy/typo handling (like 18.7); multi-word.

## 8–14. (Advantages / disadvantages / mistakes / questions / pitfalls / optimizations)

**Advantages:** ms latency (precompute + cache); read-heavy → cache-friendly; ranking done offline; trending via log aggregation; scalable (prefix sharding).
**Disadvantages/cautions:** trie memory (prune/compress); freshness lag (eventual); personalization complexity; precompute pipeline.
**Common mistakes:** ranking at query time (too slow — should precompute); no caching (read-heavy latency-critical → must cache); full-text search engine for autocomplete (overkill/slow — use precomputed top-K); unbounded trie memory (prune); ignoring latency (the whole point — 17.2).
**Interview Qs:** 🟢 What data structure + why precompute? 🟡 Precompute + cache for ms latency; ranking/freshness from logs? 🔴 Trie memory + sharding + trending updates? ⚫ Full design (trie/prefix→top-K + cache + log-aggregation freshness + sharding).
**Production pitfalls:** slow suggestions (ranking at query time / no cache); trie memory blowout (no pruning); stale suggestions (infrequent recompute); tail latency spikes (17.2).
**Optimizations:** precomputed top-K per prefix + aggressive cache/CDN; offline ranking from query-log aggregation (9.6); prefix sharding (7.3); prune/compress trie; trending via streaming windows (9.6).

## 15. Summary

**Autocomplete/typeahead** returns the **top-K most relevant completions** for a **prefix**, firing on **every keystroke** → it is **extremely latency-critical** (17.2 — must feel instant, low-ms), **read-heavy**, and **top-K-ranked** — a **precompute + cache** design distinct from full-text search (18.7). The **key signal** — latency-critical + read-heavy + top-K per prefix — dictates **precompute the top-K per prefix + cache aggressively** so the query is a **lookup, not a computation**. The **data structure** is a **trie (prefix tree)** with the **top-K completions precomputed and stored at each prefix node** → a query **walks to the prefix node and returns its precomputed top-K** (**O(prefix length)** + constant — extremely fast, **no ranking at query time**); an equivalent is a **prefix→top-K map in a KV store/cache** (6.6 — simpler to shard). **Precompute + cache** is the strategy: **precompute top-K offline** (batch — 9.7) and **serve from an in-memory cache** (Redis/CDN — 6.6/18.4 — highly cacheable, read-heavy, popular prefixes → near-100% hit) for **~ms latency** — doing **zero ranking at query time** (latency — 17.2 — drives everything). **Ranking + freshness** come from **query logs**: user queries are **logged as events** (18.1) → **stream/batch aggregation** (9.6/9.7) computes **query frequencies** → **periodically recompute top-K per prefix** and update the trie/cache, with **trending** surfacing via recent-window aggregation (9.6) — so autocomplete **learns from what people search** (a data pipeline like recommendations — 18.5), with **async/eventual** updates (10.5 — fine). **Scale** via **sharding by prefix** (7.3) + cache/replicas for reads (Part 6/7.5). **Deep dives**: **latency** (the whole design — precompute + cache + no query-time ranking + watch tail — 17.2), **trie memory** (**prune** low-popularity prefixes/completions, compress via radix tree, or prefix→top-K in KV), **freshness vs cost** (recompute frequency; streaming for trending), **personalization** (blend global + user-specific), and **correctness** (typos/fuzzy — like 18.7, multi-word). There's **no extreme bottleneck** — precompute + cache dissolves the latency concern; **trie memory + the freshness pipeline** are the interesting parts. Autocomplete = **precomputed top-K per prefix (trie / prefix→top-K) + aggressive caching (Part 6) for ms latency (17.2) + ranking/freshness from query-log aggregation (9.6/18.1) + prefix sharding (7.3)** — precompute + cache is the whole game.
