## 2. Problem statement

Design a **chat system** (WhatsApp/Messenger-style): **1:1 and group messaging**, **real-time delivery** to online users, **offline delivery** when a recipient is away, **delivery states** (sent/delivered/read), **ordering**, and **presence** (online status), at the scale of **hundreds of millions of concurrent connections**. This is the **applied version of 18.8** (Chat at Scale) driven through the interview framework.

---

## 3. The design (framework — 1.3.1)

### 3.1 Requirements

`[BP]`
- **Functional:** send/receive **1:1** + **group** messages in real time; **offline delivery** (deliver on reconnect + push notification); **delivery states** (sent → delivered → read); message **history**; **presence** (online/last-seen).
- **Non-functional:** **real-time** (low latency, push), **massive concurrent connections** (C10M — 17.3), **reliable + ordered + no-duplicates** delivery, **highly available**, **scalable storage**.
- `[BP]` **Key signals** (18.8): **real-time push** (persistent connections), **connection-scale** (C10M), **reliable ordered delivery**, **offline handling**. → the 18.8 architecture.

### 3.2 HLD — the 18.8 architecture applied

`[BP]` (Full treatment in 18.8; here applied to the interview):
- **Connection layer** (18.8 §3.2, 3.2.5/17.3): clients hold **persistent WebSocket connections** to a fleet of **event-driven connection gateways** (17.3 — C10M); a **connection registry** tracks which gateway holds each user.
- **Message flow (store + route)** (18.8 §3.3): sender → gateway → **message service** → **(a) store** the message (wide-column — 18.2) **+ (b) look up the recipient's gateway** (registry) → **push** to their connection; if **offline** → **store + queue + push notification** → deliver on reconnect. Often via a **pub-sub bus** (9.1) between the service and gateways.
- **Storage** (18.8 §3.5, 18.2): wide-column — **partition by conversation, sort by time** (high write throughput — LSM); shared conversation log and/or per-recipient inbox.
- `[BP]` See 18.8 for the full architecture; the design is the **applied 18.8**.

### 3.3 Delivery guarantees + ordering + dedup (18.8 §3.6)

`[CS]` `[BP]`
- **Delivery states:** **sent** (server got it) → **delivered** (recipient device got it) → **read** — acknowledgment events flowing back (the checkmarks).
- **Ordering** (9.5): messages ordered **per conversation** via a **server-assigned sequence number** (careful with clock skew — 8.1.2).
- **Idempotency/dedup** (11.5): **client-generated message IDs** + server dedup → resends (flaky mobile networks) don't **duplicate** → **exactly-once effect** (9.4/11.5).
- **Reliable delivery** (9.4): at-least-once (store + retry until acked) + dedup by ID.
- `[BP]` **Reliable, ordered, deduplicated per-conversation delivery** — via stored messages + acks + per-conversation sequence + idempotent IDs (18.8).

### 3.4 Group chat + fan-out (18.8 §3.4)

`[BP]`
- **1:1:** route to the one recipient (§3.2).
- **Small groups (fan-out-on-write):** write/push to each member (18.8/19.1.5 push).
- **Large groups (fan-out-on-read/hybrid):** members read from a **shared conversation log** (avoid millions of writes — the celebrity/broadcast problem — 7.4/19.1.5).
- `[BP]` **The same fan-out tradeoff** as feed (19.1.5) + chat (18.8): on-write for small, on-read/hybrid for huge groups.

### 3.5 Presence + offline (18.8 §3.7)

`[BP]`
- **Presence** (18.8 §3.7): online/offline via connection state + heartbeats (8.3.5); showing status to many contacts is a **high-fan-out** problem → **batch/throttle/approximate** (eventual — 10.5); pub-sub to interested subscribers only.
- **Offline delivery** (§3.2, 19.1.4): store + queue + **push notification** (via a notification system — 19.1.4) → deliver on reconnect.
- `[BP]` Presence is deceptively expensive (fan-out); offline delivery reuses store-and-forward + push notifications (19.1.4).

### 3.6 Deep dives + bottlenecks

`[BP]`
- **Connection scale (C10M — 17.3):** event-driven gateways + connection registry — the defining challenge (18.8 §3.2).
- **Delivery/ordering/dedup** (§3.3): the correctness concern.
- **Group fan-out** (§3.4): on-write vs on-read by size.
- **Presence fan-out** (§3.5): batch/throttle.
- **Storage** (18.2): partition by conversation; watch **hot conversations/channels** (7.4).
- **E2E encryption** (optional — 15.3): route ciphertext (18.8 §3.7).
- **Bottleneck:** connection layer (C10M — event-driven gateways) + presence fan-out + hot conversations. Storage/routing scale via wide-column + pub-sub.
- `[BP]` **The lesson:** chat = **applied 18.8** — persistent connections (C10M) + connection registry + store-and-route + wide-column storage + reliable/ordered/deduped delivery + fan-out (feed-style) + presence + offline (push notifications — 19.1.4).

## 5. Real-World Analogy

(See 18.8 §5 for the full analogy — a planet-scale postal + switchboard hybrid.) In brief: **everyone keeps an open phone line** (persistent connections) to super-efficient **switchboards** (event-driven gateways) that juggle hundreds of thousands of lines each; a **central directory** knows which switchboard holds whose line (registry); a message is **written down + rung through** (store + route), or **left in a mailbox with a doorbell** if the line's down (offline + push notification); messages are **tracked** (sent/delivered/read), kept **in order per conversation**, and **deduplicated by ticket number** (idempotent IDs); groups either **copy to each member** (small) or use a **shared channel bulletin** (huge); and the **"who's online" board** is deceptively expensive (batch/throttle).

## 7. Implementation Details

- **Persistent connections + gateways + registry** (18.8 §3.2, 3.2.5/17.3): event-driven gateways (C10M); registry tracks connections.
- **Store + route via pub-sub** (18.8 §3.3, 9.1): persist (wide-column — 18.2) + route to recipient's gateway + push; offline → queue + push notification (19.1.4) + deliver on reconnect.
- **Delivery/ordering/dedup** (§3.3, 9.4/9.5/11.5): sent/delivered/read acks; per-conversation server sequence; client IDs + dedup.
- **Group fan-out** (§3.4, 18.8): on-write small / on-read-hybrid huge (7.4).
- **Presence** (§3.5, 18.8): heartbeats (8.3.5) + batch/throttle/approximate (10.5).
- **Storage** (18.2): partition by conversation; watch hot conversations (7.4).
- **E2E** (optional — 15.3): route ciphertext.

## 8–14. (See 18.8 for full detail)

**Advantages:** real-time (push), massive connection scale (event-driven), reliable/ordered/deduped, offline delivery, scalable storage.
**Disadvantages/cautions:** connection-scale complexity; presence fan-out; huge-group fan-out; hot conversations.
**Common mistakes:** polling (not push); thread-per-connection (17.3); fan-out-on-write to huge groups; no idempotency (duplicates); naive presence fan-out; no offline store-and-forward.
**Interview Qs:** 🟢 Why persistent connections + how offline delivery? 🟡 C10M connections + routing (registry)? Reliable ordered deduped delivery? 🔴 Group fan-out + presence at scale? ⚫ Full chat design (= applied 18.8).
**Production pitfalls:** connection-scale meltdown; duplicate messages; presence storm; lost offline messages; out-of-order; hot-conversation shard.
**Optimizations:** event-driven gateways + registry; store-and-forward + pub-sub; wide-column storage; idempotent IDs + per-conversation ordering; feed-style fan-out; batch/throttle presence.

## 15. Summary

A **chat system** (WhatsApp/Messenger) provides **1:1 + group messaging** with **real-time delivery** to online users, **offline delivery** when away, **delivery states** (sent/delivered/read), **ordering**, and **presence**, at **hundreds of millions of concurrent connections** — this is the **applied version of 18.8** driven through the framework. The **key signals** (18.8): **real-time push** (persistent connections), **connection-scale** (C10M — 17.3), **reliable ordered delivery**, and **offline handling**. The **architecture** (18.8): clients hold **persistent WebSocket connections** (3.2.5) to a fleet of **event-driven connection gateways** (17.3 — thread-per-connection is impossible at C10M), with a **connection registry** tracking which gateway holds each user; message delivery is **store + route** — persist the message (**wide-column** — 18.2, partition by conversation, sort by time) **and** look up the recipient's gateway (registry) to **push** it, or if the recipient is **offline**, **store + queue + push notification** (via a notification system — 19.1.4) and deliver on reconnect (often via a **pub-sub bus** — 9.1). **Delivery guarantees**: **sent → delivered → read** acknowledgment states; **per-conversation ordering** via a **server-assigned sequence** (9.5); and **client-generated message IDs + server dedup** (11.5) so resends over flaky mobile networks don't **duplicate** (**exactly-once effect** — 9.4). **Group fan-out** uses the **same tradeoff as feed** (19.1.5) + chat (18.8): **fan-out-on-write** for small groups, **fan-out-on-read/hybrid** (shared conversation log) for huge groups (avoiding the broadcast write explosion — 7.4). **Presence** ("who's online") is deceptively expensive (high-churn + high-fan-out to contacts) → **batch/throttle/approximate** (eventual — 10.5, heartbeats — 8.3.5). The **bottleneck** is the **connection layer (C10M)** + **presence fan-out** + **hot conversations** (7.4); storage/routing scale via wide-column + pub-sub. Chat is **applied 18.8** — persistent connections + registry + store-and-route + wide-column storage + reliable/ordered/deduped delivery + feed-style fan-out + presence + offline push notifications (19.1.4). **(See 18.8 for full depth.)**
